# Scout Workflow (技能侦察兵工作流)

**Goal**: 发现外部优质技能资源，私域化定制为本仓库标准能力，并持续巡逻优化。

## 模式路由 (Mode Router)

根据用户触发词，分发到对应的步骤链：

| 触发词 | 目标模式 |
|-------|---------|
| `"启动 Skill Scout，我需要一个..."` / `"帮我找一个能...的技能"` / `"搜索一个...的 skill"` / `"scout 模式"` | → **Scout 模式** |
| `"启动巡逻模式"` / `"检查能力更新"` / `"巡逻一下现有技能"` / `"patrol 模式"` | → **Patrol 模式** |
| `"管理资源来源"` / `"添加来源"` / `"移除来源"` / `"查看来源列表"` | → **Source Registry 管理（内置层）** |
| `"添加用户来源"` / `"添加来源到用户层"` | → **Source Registry 管理 — 添加用户来源** |
| `"屏蔽内置来源"` / `"禁用 {name}"` | → **Source Registry 管理 — 屏蔽内置来源** |
| `"移除用户来源"` / `"删除用户来源"` / `"移除用户层的 {name}"` | → **Source Registry 管理 — 移除用户来源** |
| `"只看用户来源"` / `"只看内置来源"` | → **Source Registry 管理 — 按层级筛选来源列表** |

**路由规则**：
1. 解析用户输入，匹配上述触发词或语义等价表达。
2. 若无法判断模式，向用户确认意图后再分发。
3. 同一会话中可切换模式，但需完成当前步骤链或显式中断。

## Scout 模式 (发现与私域化)

**目标**：接收自然语言需求 → 搜索外部资源 → 评估选择 → 私域化改造 → 注册部署。

### 步骤链

```
parse → search → evaluate → adapt → register
```

| 步骤 | 文件 | 说明 |
|-----|------|------|
| 1. Parse (需求解析) | `step-01-parse.md` | 从自然语言中提取结构化 SearchIntent |
| 2. Search (资源搜索) | `step-02-search.md` | 在 Source Registry 来源中搜索匹配 skill |
| 3. Evaluate (评估选择) | `step-03-evaluate.md` | 详细分析候选项，用户确认改造基线 |
| 4. Adapt (私域化改造) | `step-04-adapt.md` | 收集个性化需求，调用 Skill Forge 生成文件 |
| 5. Register (注册部署) | `step-05-register.md` | 部署到 .agents/skills/ 并更新私域能力清单 |

### 执行协议
1. 按顺序执行，**不得跳步**。
2. 每个步骤开始前，阅读对应的完整步骤文件。
3. 每个步骤的输出作为下一步骤的输入。
4. 用户可在任意步骤中止流程。
5. 若本轮目标是“生成 skill”或“重写 skill 核心定义”，则 `search` 步骤必须包含实际联网检索证据；否则流程不得进入 `evaluate`。

### Scout 模式硬门槛

以下条件必须同时满足，才允许从 `search` 进入 `evaluate`：

1. 已完成 Effective Sources 计算
2. 已完成实际联网检索，不只是来源池映射
3. 已在 `search` 产物中显式记录：
   - 打开的来源链接
   - 为什么它是高相似度候选
   - 本轮如何使用该来源

若以上任一条件不满足：

- 本轮 Scout 流程必须停留在 `search`
- 不得进入 `evaluate`
- 不得生成 skill

## Patrol 模式 (巡逻与优化)

**目标**：扫描已有私域能力 → 搜索外部更优资源 → 差异分析 → 生成报告 → 辅助优化。

### 步骤链

```
scan → diff → report → optimize
```

| 步骤 | 文件 | 说明 |
|-----|------|------|
| 1. Scan (能力扫描) | `patrol-01-scan.md` | 读取私域能力清单，搜索外部相似资源 |
| 2. Diff (差异分析) | `patrol-02-diff.md` | 对比 Private Skill 与外部资源的差异 |
| 3. Report (报告生成) | `patrol-03-report.md` | 汇总优化机会，按价值排序生成报告 |
| 4. Optimize (优化执行) | `patrol-04-optimize.md` | 备份原文件，执行优化，更新版本 |

### 执行协议
1. 按顺序执行，**不得跳步**。
2. 每个步骤开始前，阅读对应的完整步骤文件。
3. 若 Report 阶段无优化机会，标注"当前能力保持先进"并结束流程。
4. Optimize 阶段需用户逐项确认后才执行。

## Source Registry 管理

**目标**：维护外部资源来源列表，控制搜索范围和质量。支持内置层与用户层的分层管理。

**内置层配置文件**：`references/source-registry.yaml`

**用户层配置文件**：`skill-sources.yaml`

**偏好配置**：`references/user-source-preferences.yaml`

### 数据结构 (SourceRegistryEntry)

每个来源条目包含以下字段：

```yaml
- name: string          # 来源名称（唯一标识，不可重复）
  url: string           # URL 或标识（必须为合法的 HTTP/HTTPS URL）
  type: string          # 来源类型：github / marketplace / community
  priority: integer     # 优先级权重 (1-10, 10 为最高优先级)
  description: string   # 来源描述
```

用户层条目额外支持以下可选字段：

```yaml
  disabled: boolean     # 可选，设为 true 可屏蔽同名内置来源
```

### 来源分层策略（两层架构）

Source Registry 采用 **内置精选层（Built-in）→ 用户自定义层（User）** 的两层架构，实现来源的分层管理与独立维护：

#### 第一层：内置精选层（Built-in Layer）

- **路径**：`references/source-registry.yaml`（skill 内部）
- **定位**：保存全局默认来源池，仅收录通用、高质量、值得长期维护的核心来源
- **维护者**：skill 维护者管理，用户不应直接修改此文件
- **数据结构**：标准 `SourceRegistryEntry`

#### 第二层：用户自定义层（User Layer）

- **路径**：`skill-sources.yaml`（仓库根目录）
- **定位**：保存用户自行添加的外部资源来源，与内置精选来源独立维护
- **维护者**：用户/团队自行管理
- **数据结构**：与内置层一致的 `SourceRegistryEntry`，额外支持 `disabled: boolean` 可选字段（设为 `true` 可屏蔽同名内置来源）

#### 合并规则

- 以内置层为基础集合，遍历用户层条目进行合并
- **同名覆盖**：用户层条目与内置层同名时，用户层覆盖内置层
- **屏蔽机制**：用户层条目标记 `disabled: true` 且同名存在于内置层时，从有效集合中移除该内置来源
- **新增追加**：用户层中内置层不存在的条目，追加到有效集合并标记 `layer: user`
- **向后兼容**：用户层文件不存在时，行为与现有单层系统完全一致

#### 偏好过滤（现有机制不变）

- **路径**：`references/user-source-preferences.yaml`
- **定位**：保存用户/团队/场景级偏好，通过白名单、黑名单、类型偏好、优先级加权对来源做二次收敛

#### 配置文件引用列表

| 文件 | 路径 | 用途 |
|------|------|------|
| 内置来源注册表 | `references/source-registry.yaml` | 内置精选来源（Built-in Layer） |
| 用户来源注册表 | `skill-sources.yaml` | 用户自定义来源（User Layer） |
| 用户偏好配置 | `references/user-source-preferences.yaml` | 白名单/黑名单/类型偏好/优先级加权 |

执行搜索时，应先通过 Source_Registry_Loader 从两层加载并合并来源，计算 **Effective Sources（有效来源集合）**，再应用偏好过滤，最终进入实际检索流程。

### 操作一：添加来源（内置层）

**触发词**：`"添加来源"` / `"新增资源来源"` / `"注册一个新来源"`

**输入**：用户提供来源信息（name、url、type、priority、description）

**执行流程**：

1. **收集信息**：向用户收集以下必填字段：
   - `name`：来源名称（字符串，不可为空）
   - `url`：来源 URL（必须为合法的 HTTP/HTTPS URL）
   - `type`：来源类型（仅限 `github` / `marketplace` / `community`）
   - `priority`：优先级权重（整数，范围 1-10）
   - `description`：来源描述（字符串，不可为空）

2. **URL 格式校验**：
   - 检查 URL 是否以 `http://` 或 `https://` 开头。
   - 检查 URL 是否包含合法的域名部分。
   - ❌ 若格式无效 → 拒绝添加，向用户提示：`"URL 格式无效，请提供以 http:// 或 https:// 开头的合法 URL。"`

3. **名称唯一性校验**：
   - 读取 `source-registry.yaml`，检查是否已存在同名条目。
   - ❌ 若名称重复 → 提示用户：`"来源名称 '{name}' 已存在，请使用其他名称或先移除已有条目。"`

4. **可访问性验证**：
   - 尝试访问该 URL，确认来源当前可达。
   - ✅ 若可访问 → 继续执行步骤 5。
   - ⚠️ 若不可访问 → 向用户发出警告：`"该来源当前不可访问（原因：{错误信息}）。是否仍要添加？"` 用户确认后继续，否则中止。

5. **写入注册表**：
   - 将新条目追加到 `source-registry.yaml` 的 `sources` 列表中。
   - 保持 YAML 格式规范，确保缩进和结构一致。

6. **确认反馈**：
   - 向用户展示已添加的条目信息，确认操作成功。

**输出示例**：

```
✅ 来源已添加：
  名称: Awesome AI Agents
  URL: https://github.com/e2b-dev/awesome-ai-agents
  类型: github
  优先级: 8
  描述: 精选 AI Agent 框架与工具合集
```

### 操作二：移除来源（内置层）

**触发词**：`"移除来源"` / `"删除资源来源"` / `"移除 {name}"`

**输入**：用户提供要移除的来源名称（name）

**执行流程**：

1. **读取注册表**：
   - 读取 `source-registry.yaml`，加载当前来源列表。
   - ❌ 若配置文件不存在或格式损坏 → 执行**配置文件恢复流程**（见错误处理）。

2. **查找目标条目**：
   - 在 `sources` 列表中按 `name` 字段精确匹配目标条目。
   - ❌ 若未找到匹配条目 → 提示用户：`"来源 '{name}' 不在注册表中。"` 并展示当前来源列表供参考。

3. **确认移除**：
   - 向用户展示即将移除的条目详情，请求确认：`"确认移除以下来源？\n  名称: {name}\n  URL: {url}\n  类型: {type}"`
   - 用户确认后继续，否则中止。

4. **执行移除**：
   - 从 `sources` 列表中删除该条目。
   - 将更新后的列表写回 `source-registry.yaml`。

5. **确认反馈**：
   - 向用户确认移除成功，展示剩余来源数量。

**输出示例**：

```
✅ 来源已移除：Awesome AI Agents
   当前注册表中剩余 2 个来源。
```

### 操作三：添加用户来源（用户层）

**触发词**：`"添加用户来源"` / `"添加来源到用户层"`

**输入**：用户提供来源信息（name、url、type、priority、description）

**执行流程**：

1. **收集信息**：向用户收集以下必填字段：
   - `name`：来源名称（字符串，不可为空）
   - `url`：来源 URL（必须为合法的 HTTP/HTTPS URL）
   - `type`：来源类型（仅限 `github` / `marketplace` / `community`）
   - `priority`：优先级权重（整数，范围 1-10）
   - `description`：来源描述（字符串，不可为空）

2. **URL 格式校验**：
   - 检查 URL 是否以 `http://` 或 `https://` 开头。
   - 检查 URL 是否包含合法的域名部分。
   - ❌ 若格式无效 → 拒绝添加，向用户提示：`"URL 格式无效，请提供以 http:// 或 https:// 开头的合法 URL。"`

3. **名称唯一性校验（用户层内）**：
   - 读取 `skill-sources.yaml`（若存在），检查用户层内是否已存在同名条目。
   - ❌ 若用户层内名称重复 → 提示用户：`"来源名称 '{name}' 已存在于用户层，请使用其他名称或先移除已有条目。"`

4. **内置层同名检测**：
   - 读取 `references/source-registry.yaml`，检查内置层是否存在同名条目。
   - ⚠️ 若内置层存在同名条目 → 向用户发出提示：`"内置层中已存在同名来源 '{name}'。添加到用户层后将覆盖内置层的该来源。是否继续？"` 用户确认后继续，否则中止。

5. **可访问性验证**：
   - 尝试访问该 URL，确认来源当前可达。
   - ✅ 若可访问 → 继续执行步骤 6。
   - ⚠️ 若不可访问 → 向用户发出警告：`"该来源当前不可访问（原因：{错误信息}）。是否仍要添加？"` 用户确认后继续，否则中止。

6. **检查用户层文件**：
   - 检查 `skill-sources.yaml` 文件是否存在。
   - 若文件不存在 → 创建包含注释说明的空模板文件（使用 UserRegistryTemplate 格式）：

```yaml
# Skill Sources - 用户自定义技能搜索来源
# 本文件存放用户自行添加的外部资源来源，与内置精选来源独立维护
# 同名条目会覆盖内置层；添加 disabled: true 可屏蔽内置来源
#
# 数据结构与内置层一致 (SourceRegistryEntry):
#   - name: string          # 来源名称（同名则覆盖内置层）
#     url: string           # URL 或标识
#     type: string          # github / marketplace / community
#     priority: integer     # 优先级权重 (1-10, 10 最高)
#     description: string   # 来源描述
#     disabled: boolean     # 可选，设为 true 可屏蔽同名内置来源
#
# 示例：
#   - name: "My Custom Source"
#     url: "https://github.com/user/repo"
#     type: github
#     priority: 7
#     description: "我的自定义来源"

sources: []
```

   - 初始化完成后向用户展示两层架构简要说明：
     `"已初始化用户层来源配置文件 skill-sources.yaml。\n  · 内置层（references/source-registry.yaml）：由 skill 维护者管理的精选来源\n  · 用户层（skill-sources.yaml）：您自行管理的自定义来源\n  用户层条目会覆盖同名内置来源，也可通过 disabled: true 屏蔽内置来源。"`

7. **写入用户注册表**：
   - 将新条目追加到 `skill-sources.yaml` 的 `sources` 列表中。
   - 保持 YAML 格式规范，确保缩进和结构一致。

8. **确认反馈**：
   - 向用户展示已添加的条目信息，明确标注"已添加到用户层"。

**输出示例**：

```
✅ 来源已添加到用户层：
  名称: My Custom Source
  URL: https://github.com/user/repo
  类型: github
  优先级: 7
  描述: 我的自定义来源
  层级: user
```

### 操作四：屏蔽内置来源

**触发词**：`"屏蔽内置来源"` / `"禁用 {name}"`

**输入**：用户提供要屏蔽的内置来源名称（name）

**执行流程**：

1. **读取内置层**：
   - 读取 `references/source-registry.yaml`，加载内置层来源列表。
   - ❌ 若配置文件不存在或格式损坏 → 执行**配置文件恢复流程**（见错误处理）。

2. **确认目标存在于内置层**：
   - 在内置层 `sources` 列表中按 `name` 字段精确匹配目标条目。
   - ❌ 若未找到匹配条目 → 提示用户：`"来源 '{name}' 不在内置层中。"` 并展示内置层来源列表供参考。

3. **检查用户层是否已有同名条目**：
   - 读取 `skill-sources.yaml`（若存在），检查是否已存在同名条目。
   - ⚠️ 若已存在同名条目且 `disabled: true` → 提示用户：`"来源 '{name}' 已被屏蔽，无需重复操作。"`
   - ⚠️ 若已存在同名条目且非 disabled → 提示用户：`"用户层中已存在同名来源 '{name}'（非屏蔽条目）。是否将其替换为屏蔽条目？"` 用户确认后继续，否则中止。

4. **检查用户层文件**：
   - 若 `skill-sources.yaml` 文件不存在 → 按"操作三"步骤 6 的初始化流程创建。

5. **写入屏蔽条目**：
   - 在 `skill-sources.yaml` 的 `sources` 列表中添加同名条目，设置 `disabled: true`：

```yaml
- name: "{name}"
  url: "{原内置条目的 url}"
  type: "{原内置条目的 type}"
  priority: 0
  description: "已屏蔽的内置来源"
  disabled: true
```

6. **确认反馈**：
   - 向用户确认屏蔽成功。

**输出示例**：

```
✅ 内置来源已屏蔽：{name}
   该来源将不再出现在 Effective Sources 中。
   如需恢复，请使用「移除用户来源」操作移除对应的屏蔽条目。
```

### 操作五：移除用户来源

**触发词**：`"移除用户来源"` / `"删除用户来源"` / `"移除用户层的 {name}"`

**输入**：用户提供要移除的用户层来源名称（name）

**执行流程**：

1. **读取用户层注册表**：
   - 读取 `skill-sources.yaml`，加载用户层来源列表。
   - ❌ 若文件不存在 → 提示用户：`"用户层来源配置不存在，无可移除的条目。"`
   - ❌ 若文件存在但格式损坏 → 提示用户：`"用户层来源配置格式损坏，请手动检查 skill-sources.yaml。"`

2. **查找目标条目**：
   - 在用户层 `sources` 列表中按 `name` 字段精确匹配目标条目。
   - ❌ 若未找到匹配条目 → 提示用户：`"来源 '{name}' 不在用户层中。"` 并展示用户层来源列表供参考。

3. **区分条目类型并确认**：

   **情况 A：用户自定义条目（无 `disabled` 字段或 `disabled: false`）**
   - 向用户展示即将移除的条目详情，请求确认：`"确认移除以下用户来源？\n  名称: {name}\n  URL: {url}\n  类型: {type}"`
   - 用户确认后继续，否则中止。

   **情况 B：屏蔽条目（`disabled: true`）**
   - 读取内置层 `references/source-registry.yaml`，检查是否存在同名内置来源。
   - 向用户展示提示：`"来源 '{name}' 是一个屏蔽条目。移除后将恢复内置层的同名来源。是否继续？"`
   - 若内置层存在同名来源 → 展示该内置来源的详情供参考。
   - 用户确认后继续，否则中止。

4. **执行移除**：
   - 从用户层 `sources` 列表中删除该条目。
   - 将更新后的列表写回 `skill-sources.yaml`。

5. **确认反馈**：

   **情况 A 反馈**：
   ```
   ✅ 用户来源已移除：{name}
      用户层剩余 {N} 个条目。
   ```

   **情况 B 反馈**：
   ```
   ✅ 屏蔽条目已移除：{name}
      内置层的同名来源已恢复生效。
      用户层剩余 {N} 个条目。
   ```

### 操作六：查询来源列表（合并后展示）

**触发词**：`"查看来源列表"` / `"列出所有来源"` / `"当前有哪些资源来源"`

**筛选触发词**：`"只看用户来源"` / `"只看内置来源"`

**输入**：无（直接读取配置文件）；可选层级筛选参数

**执行流程**：

1. **加载两层来源**：
   - 读取内置层 `references/source-registry.yaml`，加载内置来源列表，为每个条目标注 `layer: built-in`。
   - ❌ 若内置层配置文件不存在或格式损坏 → 执行**配置文件恢复流程**（见错误处理）。
   - 尝试读取用户层 `skill-sources.yaml`。
   - 若用户层文件不存在 → 静默跳过，仅使用内置层。
   - 若用户层文件存在但 YAML 解析失败 → 向用户展示警告，仅使用内置层。
   - 若用户层文件存在且解析成功 → 为每个条目标注 `layer: user`，进入合并流程。

2. **合并两层来源**：
   - 以内置层为基础集合，遍历用户层条目：
     - 若条目标记 `disabled: true` 且同名条目存在于内置层 → 将内置条目标注状态为"已屏蔽"。
     - 若同名条目存在于内置层（非 disabled）→ 将内置条目标注状态为"被用户层覆盖"，用用户层条目替代并标注 `layer: user`。
     - 若为新条目（内置层中无同名）→ 追加到列表，标注 `layer: user`。
   - 未被覆盖或屏蔽的内置条目标注状态为"生效"。

3. **按层级筛选（可选）**：
   - 若用户触发词为 `"只看用户来源"` → 仅展示 `layer: user` 的条目（不含屏蔽条目）。
   - 若用户触发词为 `"只看内置来源"` → 仅展示 `layer: built-in` 的条目（含被屏蔽和被覆盖的条目，标注其状态）。
   - 若无筛选条件 → 展示合并后的完整列表。

4. **按优先级排序**：
   - 将来源列表按 `priority` 字段**降序排列**（优先级高的排在前面）。

5. **格式化展示**：
   - 以表格形式向用户展示排序后的来源列表。
   - 包含字段：名称、类型、优先级、层级、状态、描述。

**输出示例（完整列表）**：

```

| # | 名称 | 类型 | 优先级 | 层级 | 状态 | 描述 |
|---|------|------|--------|------|------|------|
| 1 | Agent Skills | community | 10 | built-in | 生效 | Agent Skills 开放标准官方组织 |
| 2 | GitHub Spec Kit | github | 9 | user | 生效 | GitHub 官方开源的 SDD 工具包 |
| 3 | SkillHub | marketplace | 8 | user | 生效 | 面向多代理的技能市场 |
| 4 | OldSource | github | 7 | built-in | 被用户层覆盖 | （已被用户层同名条目覆盖） |
| 5 | DeprecatedSource | github | 6 | built-in | 已屏蔽 | （被用户层屏蔽） |

📊 统计：内置 4 个, 用户 8 个, 屏蔽 1 个, 合计生效 11 个
```

**输出示例（只看用户来源）**：

```

| 2 | SkillHub | marketplace | 8 | 面向多代理的技能市场 |
```

**输出示例（只看内置来源）**：

**空列表处理**：若合并后的 `sources` 列表为空，向用户提示：`"当前注册表为空，请使用「添加来源」或「添加用户来源」操作添加资源来源。"`

### 错误处理

#### 1. URL 格式无效

- **触发条件**：用户提供的 URL 不以 `http://` 或 `https://` 开头，或不包含合法域名。
- **处理方式**：拒绝添加操作，向用户提示正确的 URL 格式要求。
- **示例消息**：`"❌ URL 格式无效：'{url}'。请提供以 http:// 或 https:// 开头的合法 URL，例如：https://github.com/user/repo"`

#### 2. 来源不可访问

- **触发条件**：URL 格式合法但无法访问（网络超时、404、DNS 解析失败等）。
- **处理方式**：向用户发出警告，说明不可访问的原因，询问是否仍要添加。不强制阻止，由用户决定。
- **示例消息**：`"⚠️ 该来源当前不可访问（HTTP 404 Not Found）。是否仍要将其添加到注册表？"`

#### 3. 移除不存在的条目

- **触发条件**：用户请求移除的来源名称在对应注册表中不存在。
- **处理方式**：提示该来源不存在，并展示对应层级的来源列表供用户核对名称。
- **内置层示例**：`"❌ 来源 '{name}' 不在内置层注册表中。当前内置来源有：{来源名称列表}。请核对名称后重试。"`
- **用户层示例**：`"❌ 来源 '{name}' 不在用户层中。当前用户来源有：{来源名称列表}。请核对名称后重试。"`

#### 4. 配置文件损坏（内置层）

- **触发条件**：`source-registry.yaml` 文件存在但 YAML 格式解析失败，或文件不存在。
- **恢复流程**：
  1. 尝试读取文件内容，若 YAML 解析失败，记录错误信息。
  2. 检查是否存在备份文件（`source-registry.yaml.bak`）。
  3. ✅ 若备份存在且格式正确 → 从备份恢复，提示用户：`"配置文件已损坏，已从备份恢复。"`
  4. ❌ 若无可用备份 → 创建包含空 `sources` 列表的默认配置文件，提示用户：`"配置文件已损坏且无可用备份，已创建空的默认配置。请重新添加资源来源。"`
- **默认配置模板**：

```yaml
# Source Registry - 外部资源来源注册表
# 由系统自动恢复创建

sources: []
```

#### 5. 用户层文件错误

- **触发条件**：`skill-sources.yaml` 文件存在但 YAML 格式解析失败。
- **处理方式**：记录错误日志（说明解析失败原因），跳过用户层，回退到仅使用内置层，向用户展示警告信息。
- **示例消息**：`"⚠️ 用户层来源配置解析失败（原因：{错误信息}），已回退到仅使用内置层。请检查 skill-sources.yaml 的 YAML 格式。"`

#### 6. 用户层文件不存在

- **触发条件**：执行用户层写入操作时 `skill-sources.yaml` 文件不存在。
- **处理方式**：自动创建包含注释说明的空模板文件（UserRegistryTemplate），然后继续执行操作。

#### 7. 屏蔽目标不存在于内置层

- **触发条件**：用户请求屏蔽的来源名称不存在于内置层。
- **处理方式**：提示该来源不在内置层中，展示内置层来源列表供参考。
- **示例消息**：`"❌ 来源 '{name}' 不在内置层中，无法执行屏蔽操作。当前内置来源有：{来源名称列表}。"`

## 步骤架构原则

- **Micro-Steps**: 严格遵循 `step-*.md` / `patrol-*.md` 文件定义的原子操作。每个步骤文件是自包含的指令集，包含目标、动作、输入/输出格式和错误处理。
- **Isolation**: 内存中只加载当前步骤文件。步骤间通过文件化的中间产物（YAML/Markdown）传递数据，不依赖会话记忆。

## 初始化
1. 识别用户意图，根据模式路由表分发到对应模式。
2. Scout 模式：阅读 `step-01-parse.md`，开始需求解析。
3. Patrol 模式：阅读 `patrol-01-scan.md`，开始能力扫描。
4. Source Registry 管理（内置层）：直接读取 `references/source-registry.yaml` 执行操作。
5. Source Registry 管理（用户层）：读取 `skill-sources.yaml` 执行操作（若文件不存在，按需初始化）。
6. Source Registry 管理（合并查询）：按两层加载流程合并后展示。
