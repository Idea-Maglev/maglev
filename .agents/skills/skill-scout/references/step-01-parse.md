# Step 1: Parse（需求解析）

## 目标

从用户的自然语言需求描述中提取结构化的搜索意图（SearchIntent），为后续资源搜索提供精确的输入参数。

## 动作

### 1. 提取搜索意图

从用户的自然语言描述中识别并提取以下关键要素：

- **能力类型 (capability_type)**：用户需要的能力种类，如"代码生成"、"文档整理"、"测试自动化"等。
- **目标场景 (target_scenario)**：该能力的具体应用场景，如"SQL 自动生成"、"API 文档同步"等。
- **关键约束 (constraints)**：用户提出的限制条件列表，如技术栈要求、性能要求、兼容性要求等。

### 2. 检测缺失信息

检查提取结果的完整性，若以下关键字段缺失，生成针对性的澄清问题：

| 缺失字段 | 澄清问题示例 |
|---------|------------|
| `capability_type` | "你需要的是哪种类型的能力？比如代码生成、文档处理、流程自动化？" |
| `target_scenario` | "这个能力具体用在什么场景下？能举个例子吗？" |
| `constraints`（可选但建议补充） | "有没有什么特殊要求或限制？比如技术栈、语言偏好、集成需求？" |

**规则**：`capability_type` 和 `target_scenario` 为必填项。若两者均缺失，视为需求过于模糊，触发错误处理流程。

### 3. 记录原始描述

将用户的原始自然语言描述完整保留在 `raw_description` 字段中，确保可追溯。

## 输出格式

解析完成后，生成以下 SearchIntent 结构（YAML 格式）：

```yaml
search_intent:
  capability_type: string       # 能力类型，如 "代码生成"、"文档整理"
  target_scenario: string       # 目标场景，如 "SQL 自动生成"
  constraints:                  # 关键约束列表
    - string
  raw_description: string       # 原始自然语言描述
  confirmed: false              # 初始为 false，用户确认后改为 true
```

## 交互流程

### 正常流程

1. **呈现解析结果**：将提取的 SearchIntent 以可读格式展示给用户，例如：

   AI: "我从你的描述中解析出以下搜索意图：
   - 能力类型：代码生成
   - 目标场景：SQL 自动生成
   - 关键约束：支持 PostgreSQL、需要参数化查询
   
   请确认是否准确，或告诉我需要调整的地方。"

2. **用户确认**：等待用户确认或修改。
   - 用户确认 → 将 `confirmed` 标记为 `true`，进入下一步骤。
   - 用户要求修改 → 根据反馈更新 SearchIntent，重新呈现。

3. **标记确认**：确认后的 SearchIntent 作为 `step-02-search.md` 的输入。

### 信息补充流程

当检测到缺失信息时：

AI: "我注意到你的描述中缺少一些关键信息。为了更精准地搜索，请补充：
1. [针对性澄清问题]"

用户补充后，重新执行提取流程，直到 SearchIntent 结构完整。

## 错误处理

| 错误场景 | 处理策略 |
|---------|---------|
| 需求过于模糊（无法识别能力类型和目标场景） | 向用户说明无法理解当前描述，请求重新描述或提供更具体的上下文。例如："你的描述比较宽泛，我无法确定具体需要什么能力。能否举一个具体的使用场景？比如'我想要一个能自动生成 SQL 查询的工具'。" |
| 需求描述为空或无意义输入 | 提示用户提供有效的需求描述，并给出示例引导。 |
| 用户多次修改仍无法确认 | 汇总当前理解，逐项与用户确认，确保每个字段都得到明确回复。 |

## 状态流转条件

- 当 SearchIntent 的 `capability_type` 和 `target_scenario` 均非空，且 `confirmed: true` 时，转入 `step-02-search.md`。
