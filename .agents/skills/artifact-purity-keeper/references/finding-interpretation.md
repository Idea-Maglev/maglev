# Finding Interpretation — 命中解读规则

扫描器是辅助识别, 不是门禁. 同一个 finding 在不同语境下可能是真污染, 也可能是合理的协议引用. 本文档给出判断框架.

## 核心问题

每个 finding 都问 3 个问题:

1. **这个文件的受众是谁?** (`session` / `handoff` / `external`)
2. **finding 内容是协议命名 / 公开能力名 / 稳定术语吗?** 还是 spec 内部编号 / 会话流水?
3. **删掉它读者会困惑吗?**

## 按规则解读

### `dialog-reference` (对话指代, hard)

- 出现在 external 面: **必修**
- 没有合理"协议引用"语境
- 修复: 改成第三人称客观陈述

### `iteration-tag` (迭代叙事, hard)

- 出现在 external 面: **必修**
- 极个别情况下 `v2.0` 这种产品版本号是合法的, 不应触发 (规则已限定为 `v\d+(\.\d+)?\s*\((fix|修订|重写)\)` 形式)
- 修复: 删除括号内迭代标签, 仅保留版本号本身

### `collab-narrative` (协作叙事, hard)

- 出现在 external 面: **必修**
- 例外: 历史归档/复盘类文件的"执行经验"段落允许（这类文件的受众与产生时一致, 调用方应将其标记为非 external）
- 修复: 改成"现状是什么 + 为什么"两句, 不写"如何到达现状"

### `process-code` (过程编号, soft)

最容易判错, 重点关注:

- **真污染**: 内部 spec/AC/D 编号或 Phase 编号出现在对外文件. 例: `(AC-F12-9)`, `(D27)`, `Phase 3`
- **合理协议引用**: 公开协议中稳定的步骤名（带主语限定的、协议命名空间内的稳定符号）. 例:
  - "<能力名> Step 5" — 当 Step 5 是该能力对外协议中的稳定步骤名时, 引用合理
  - "Step 1-6" 这种泛指性流程描述, 但若指向特定 spec 的 Step 编号则是污染
- **判断技巧**: finding 旁边是否有"主语限定"? 有（绑定到稳定能力命名空间）→ 协议引用; 无（裸 Step 编号）→ 污染

### `relative-time` (时间相对词, soft)

- 出现在 external 面: **建议改写**
- 例外: 写作指南/方法论类文档中, 用作泛指语义时可保留（如指南讲"提炼本轮真正改变了哪些现状", "本轮"在指南语境是泛指任意一轮, 而非特定一次）
- 修复: 改成"该轮"/"每轮"/"本次"等去时间锚定的写法

### `change-date-narrative` (变更日期叙事, soft)

- 出现在 external 面: **必修**
- "2026-04-25 取代废弃的 OldFooBar" 是典型反例（带变更日期 + 废弃叙事）
- 修复: 改述为当前事实, 如 "X → Y"，去掉日期与"取代/废弃"动词

### `narrative-shift` (状态变迁叙事, soft)

- 出现在 external 面: **必修**
- 把"变更日志/状态变迁过程"塞进了"稳定文档", 暗示读者了解前一状态
- 典型反例:
  - "它不再只是 X，而是 Y" → 应直接写 "它是 Y"
  - "已切换为 <new-name>" → 应直接写 "运行名称: <new-name>"
  - "仍保留 <legacy-entry> 兼容入口" → 应直接写 "兼容入口: <legacy-entry>"
  - "已经进入多主线收口阶段" → 删除或改述成稳定能力描述
- 修复要点: 去掉时态副词（不再/已/仍/现已）, 把变迁描述改写成现状陈述

### `confirm-phrase` (确认句式, info)

- 仅提示, 通常作者会自然意识到

## 实战判断流程

```
finding 出现
  ↓
文件是 external 面?
  ├── 否 (session/handoff) → 通常无需处理 (除非超出该层级容忍度)
  └── 是 → 继续
       ↓
       finding 内容是协议命名/稳定术语?
         ├── 是 → 标记 "协议引用, 保留", 写处置说明
         └── 否 → 真污染, 修复
              ↓
              hard 级 → 必修
              soft 级 → 建议修, 由作者决定
              info 级 → 仅提示
```

## 输出模板

每次扫描后, 给作者一份处置清单:

```markdown
## 扫描结果处置

文件: `<path>`  受众层级: `<external|handoff|session>`

### 已修复 (hard)
- L<N>: `<match>` → 改为 `<new>`  (理由: <reason>)

### 标记为协议引用, 保留 (soft)
- L<N>: `<match>`  (理由: 公开协议命名 / 稳定术语)

### 建议修复但保留判断给作者 (soft/info)
- L<N>: `<match>`  (建议改为 `<suggestion>`)
```
