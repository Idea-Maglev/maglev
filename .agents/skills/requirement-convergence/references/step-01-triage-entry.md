# Step 1: Triage Entry

## 目标

把入口任务先收成一个明确问题，而不是直接进入方案设计。

## 动作

1. 识别用户当前最想解决的核心问题。
2. 判断本轮更接近：
   - 新需求
   - 结构治理
   - bug / 偏差修正
   - 存量接入
   - 验证或复查
3. 判断当前是否存在“需求产物不足以被下游稳定消费”的问题：
   - 是否需要先生成 PRD / 需求文档
   - 是否当前只靠最小总结会继续发生漂移
   - 是否下游评审、设计或协作需要一份稳定文档基线
4. 收窄本轮真正处理的对象：
   - 是某个 skill
   - 某条流程
   - 某个项目现状问题
5. 明确哪些内容只是背景，不是本轮目标。

## 判定规则

- 如果一句话里同时出现“想法 + 实施 + 验证”，优先回到最前置的问题定义。
- 如果对象不明确，优先收“本轮核心对象”，而不是先追问实施细节。
- 如果当前请求本质是“应该先做什么”，默认判为前段收敛，而不是方案设计。
- 如果当前核心诉求是“先把需求沉淀成稳定文档，否则后续会漂移或无法充分消费”，记录为 `PRD 文档输出模式候选`，但仍先完成需求收敛。

## 输出格式

至少输出：

- `入口类型`
- `本轮核心对象`
- `输出模式候选`
- `本轮不处理的背景项`

## 输出

- 一份入口问题归类
- 一份本轮核心对象判断
