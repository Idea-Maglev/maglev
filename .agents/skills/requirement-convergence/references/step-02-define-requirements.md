# Step 2: Define Requirements

## 目标

把入口问题固定成可被后续设计消费的约束面。

## 动作

1. 明确本轮 `In Scope`：
   - 这次真正要改什么
2. 明确本轮 `Out of Scope`：
   - 这次明确不改什么
3. 固定最低成功信号：
   - 做到什么算这轮成立
4. 识别仍未解决的关键未知：
   - 哪些未知会显著影响后续方案设计
5. 判断当前更适合的输出模式：
   - `spec_design`
   - `prd_document`
6. 判断最小 handoff 是否足够：
   - 若不足，明确为什么需要稳定需求文档作为额外基线

## 判定规则

- `In Scope` 必须是这轮可交付的变化，不写长期愿景。
- `Out of Scope` 必须显式写出，避免后续自然膨胀。
- 成功信号优先写“如何判断成立”，不是写“希望做得更好”。
- 关键未知只保留会改变后续设计路径的未知。
- 若仅靠最小 handoff 无法避免需求漂移，或不足以让下游稳定消费，可将输出模式固定为 `prd_document`。

## 输出格式

至少输出：

- `In Scope`
- `Out of Scope`
- `Success Signal`
- `Key Unknowns`
- `Preferred Output Mode`
- `Why Minimum Handoff Is Not Enough`（若为空则写空）

## 结构化验收标准生成

当任务复杂度需要结构化需求时（非简单修复/独立改动），在上述输出完成后额外执行：

0. **收集来源依据**：
   - 在生成 AC 前列出本轮实际消费过的来源，包含文档、会议、聊天、设计稿、代码观察、用户确认或 AI 对话摘要
   - 每个来源至少说明：来源名称、类型、用途
   - AI 对话来源只写“来自 AI 对话”的可读摘要，不保存完整原始对话
   - 若 AI 对话中出现长期有价值的思考，建议触发 `knowledge-check` 沉淀到 `docs/thinking/`，不要塞进需求正文

1. **判断是否需要结构化需求**：
   - 有跨角色协作需求 → 需要
   - 需要行为级验收标准 → 需要
   - in_scope 一句话描述完 → 不需要
   - 修复类/独立改动 → 不需要

2. **生成功能需求初版**（当需要时）：
   - 根据 in_scope 和用户描述，自动生成功能需求列表
   - 每个功能需求包含：用户故事 + 至少 1 条 AC
   - AC 格式："当 [触发条件] 时，系统应 [响应行为]" 或 "若 [前置条件]，则系统应 [响应行为]"
   - AC 编号：AC-F{N}-{M}（F = 功能需求序号，M = AC 序号），全局唯一
   - 不允许 "支持 X 功能" 式的模糊描述
   - 每条关键 AC 必须包含：
     - `criterion`：最终验收标准
     - `source_summary`：来源表达的完整含义，不能只摘孤立关键词
     - `context_judgement`：如何从完整上下文推导为该 AC；冲突、后续补充或取舍如何处理
     - `evidence`：可回查位置、会议/聊天摘要、用户确认或 AI 对话摘要
   - 最终 AC 必须能从 `source_summary + context_judgement` 推导出来；推不出来时标为关键未知或待确认，不要伪造确定性

3. **展示给用户审核**：
   - 一次性展示全部初版 AC（不逐条追问）
   - 用户可增删改
   - 修改后重新生成编号（保持连续）

4. **处理模糊输入**：
   - 如果用户描述过于模糊无法生成 AC
   - 追问"这个功能在什么条件下触发？预期什么响应？"
   - 获得澄清后再生成

## 输出

- 一份最小需求定义
- 一组成功信号
- 一组关键未知
- 一份来源依据列表
- 结构化 AC（含验收标准、来源摘要、上下文判定、证据）

## 术语表生成

当生成结构化需求文档时，在 AC 列表之后生成术语表：

1. 扫描当前功能描述和 AC 中出现的专有名词、缩写、易混淆术语
2. 为每个术语提供一句话定义
3. 如果 `specs/10_reality/glossary.md` 存在，检查是否有已定义的术语可复用
4. 新术语追加到项目级术语文件的待确认区

格式：
- **{术语}**: {定义}
