# 交互需求文档模板

> 含 UI 项目时，由 spec-designer 在 draft 阶段按此模板生成交互需求区段。
> 不含 UI 的项目跳过此模板。

## 何时生成

- 需求中包含用户界面交互（表单、导航、状态切换等）
- 需求中包含 I 系 AC（AC-I{N}-{M}）
- 用户或 PRD 描述了界面行为期望

## 何时不生成

- 纯后端服务、CLI 工具、数据管道等无 UI 项目
- 交互极简（仅一个确认弹窗），可直接在功能需求中附加 AC

## 文档结构

```markdown
# 交互需求文档

## 文档关系

- **功能需求**: `01_requirements.md`（本文档中的交互需求可引用 F 系 AC 作为前置条件）
- **交互设计**: → `02_design_interaction.md`（消费本文档的 I 系 AC）
- **技术设计**: → `02_design.md`（可能引用本文档中的可访问性和性能约束）

## 简介

{从用户交互视角描述本功能的核心体验目标：用户看到什么、怎么操作、什么感受}

## 术语表

- **{术语 A}**: {定义}
- **{术语 B}**: {定义}

## 交互需求

### 交互需求 1: {场景名称}（I-1）

**交互场景**: {用户在什么场景下进行什么交互操作}

#### 验收标准

1. `AC-I1-1` 当 {UI 状态/用户操作} 时，界面应 {视觉/行为响应}
2. `AC-I1-2` 当 {触发条件} 时，{组件} 应 {交互行为}

#### 边界情况

- {极端交互场景} → {预期处理}

### 交互需求 2: {场景名称}（I-2）

…（同上结构）
```

## AC 维度要求

每条交互需求的 AC 应覆盖以下至少一个维度：

| 维度 | 关注点 | 典型 AC 模式 |
|------|--------|-------------|
| **UI 状态** | 加载/空/错误/成功/骨架屏 | 当 {数据状态} 时，界面应显示 {状态} |
| **操作响应** | 点击/输入/手势/键盘 | 当用户 {操作} 时，界面应 {反馈} |
| **视觉约束** | 响应式断点/布局切换/动效 | 在 {屏幕尺寸} 下，{组件} 应 {布局行为} |
| **可访问性** | 键盘导航/屏幕阅读器/对比度 | 当使用 {辅助技术} 时，{元素} 应 {行为} |

## 格式规则

| 元素 | 规则 |
|------|------|
| 需求编号 | `I-{N}`，从 1 开始递增（与 F 系独立编号） |
| AC 编号 | `AC-I{N}-{M}`，全局唯一，与 F 系不冲突 |
| EARS 关键词 | 同功能需求：当…时 / 若…则 / 在…期间 / 应 |
| 交互场景 | 替代"用户故事"，直接描述用户与界面的交互上下文 |
| F 系引用 | 可引用功能 AC 作为前置条件（如"当 AC-F1-1 的登录成功后"） |
| 文档关系 | 必须包含。声明依赖和下游 |
| 术语表 | 必须包含。侧重 UI/UX 领域术语 |

## 简单交互的内嵌写法

当交互需求非常简单时（如仅一个确认弹窗），不必创建独立交互需求文档。直接在功能需求中附加交互 AC：

```markdown
### 需求 N: 删除确认（F-N）

**用户故事**: 作为用户，我希望删除操作有确认步骤，以便防止误操作。

#### 验收标准

1. `AC-FN-1` 当用户点击删除按钮时，系统应弹出确认对话框
2. `AC-FN-2` 当用户确认删除时，系统应执行删除并关闭对话框
3. `AC-FN-3` 当确认对话框显示时，界面应聚焦到"取消"按钮（安全默认）
```
