# Step 3: Check Boundaries

## 目标

确保当前检查没有误滑入其他生命周期动作。

## 动作

1. 明确当前是否有人把这一步误解成：
   - 需求归档
   - reality writeback
   - active 收口
2. 若存在这类误解，明确指出：
   - 当前对象只负责知识沉淀检查
   - 其他动作应转交 `crystallization` 或相邻对象

## 边界规则

- “需求结束了如何收口” 属于 `crystallization`
- “哪些内容写回 Reality” 属于 `crystallization`
- “哪些思考和方案还没落盘” 属于 `knowledge-check`

## 输出格式

- `boundary_result`
- `reroute_needed`
- `reroute_target`

## 输出

- 一份边界判断
- 一条必要的转交建议
