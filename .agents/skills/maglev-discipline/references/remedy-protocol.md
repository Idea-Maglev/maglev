# 补救协议 (Remedy Protocol)

> 本文件是 maglev-discipline 的核心参考资料之一。
> 包含 L0-L4 压力升级、通用 5 步方法论、7 项强制检查清单、Owner 四问、冰山法则、信心门控 6 步与体面退出协议。

---

## L0-L4 压力升级

### 分级定义

| 等级 | 触发条件 | 响应模式 | 强制输出 |
|------|---------|---------|---------|
| **L0 正常** | 默认状态 | 遵守三条红线 | `[MAGLEV-DIAGNOSIS]` 在改动前 |
| **L1 温和失望** | 第 2 次同类失败 | 切换**本质不同**的方案 | 必须列出"旧方案 vs 新方案"对比 |
| **L2 灵魂拷问** | 第 3 次同类失败 | 搜索 + 读源码 + 列 3 假设 + 反转 | 完整假设列表 + 排除证据 |
| **L3 责任升级** | 第 4 次同类失败 | 7 项检查清单全走完 | 清单逐项打勾 |
| **L4 治理边界** | 第 5 次+ / 触犯红线 | 拼命模式 → 正式声明问题边界 | 结构化失败报告 |

### 触发三源

1. **AI 自报**：`[MAGLEV L{N}] 第 {N} 次同类失败，进入 L{N} 响应`
2. **用户手动**：用户显式说 "升 L2" / "升 L3" / "走 7 项清单"
3. **integrated-validator**：review 阶段标记 "建议升 L{N}"

### "同类失败"判定

- 同一个错误信息反复出现 → 同类
- 同一个文件 / 模块 / 接口反复修改不成功 → 同类
- 换了细节但思路方向不变 → 同类
- 切到完全不同的假设和验证路径 → **不同类**，计数重置

## 通用 5 步方法论

卡壳时强制执行，不允许跳步。

### 第 1 步：闻味道

列出所有已尝试方案，找共同模式。

- 这些方案有什么共同假设？
- 哪个假设可能是错的？
- 同一思路微调参数 = 原地打转，不算新尝试

### 第 2 步：揪头发

按序执行以下动作（跳过任何一个 = 惰性）：

1. **逐字读失败信号**：完整的错误消息、stack trace、日志，不是摘要
2. **主动搜索**：用错误原文搜索 + 官方文档 + 至少 2 种不同关键词
3. **读原始材料**：源码上下文至少 50 行（不是 AI 摘要，是原文）
4. **验证前置假设**：版本 / 路径 / 权限 / 依赖 / 环境变量 — 逐一确认
5. **反转假设**：一直假设"问题在 A" → 现在假设"问题不在 A"，验证

### 第 3 步：照镜子

自检三问（诚实作答）：

- 我是否在重复？同一个方向换参数不叫"新方案"
- 我是否该搜索却没搜？错误信息有没有 Google / grep 过？
- 我是否忽略了最简单的可能？有没有 typo、路径错误、版本不对？

### 第 4 步：执行新方案

- 必须与之前尝试的方案**本质不同**（换了假设方向，不是换了参数）
- 必须有**明确的验证标准**：跑什么命令？看什么输出？什么算成功？
- 必须先写出"如果这个方案也失败，我会看到什么信号"

### 第 5 步：复盘

解决后做三件事：

1. **同类检查**：同模块同类位置还有没有类似问题？
2. **完整性验证**：修复是否引入了新问题？回归测试过了吗？
3. **预防措施**：能不能加个检查 / 测试 / 文档防止同类再发？

## 7 项强制检查清单（L3 必做）

进入 L3 时逐项完成，不允许跳过：

| # | 检查项 | 动作 |
|---|--------|------|
| 1 | 错误信号完整阅读 | 贴出完整错误文本（不截断） |
| 2 | 搜索 3 种关键词 | 列出关键词 + 搜索结果摘要 |
| 3 | 源码上下文 ≥50 行 | 贴出具体文件 + 行号范围 |
| 4 | 前置假设列表 | 列出所有假设 + 每个的验证状态 |
| 5 | 反转假设实验 | 描述反转的假设 + 验证结果 |
| 6 | 环境一致性 | 版本 / 路径 / 权限 / 依赖全部确认 |
| 7 | 最简可能排除 | typo / 路径 / 拼写 / 导入路径逐一确认 |

完成后仍失败 → 进入 L4。

## L4 拼命模式

7 项清单走完后仍未解决，进入最后冲刺：

### 拼命模式动作清单

1. **扩大搜索面**：跨模块 / 跨仓库 / 跨语言搜索相关关键词
2. **读上游源码**：不是文档，是依赖库的实际实现
3. **最小复现**：剥离所有无关因素，造一个最小可复现的场景
4. **反向验证**：让正确行为发生需要什么条件？从终态反推
5. **求助准备**：如果最终要交接，准备好结构化失败报告（让接手人 0 成本进入）

### 退出 L4 的两种路径

- **路径 A — 解决了**：完成后走信心门控 6 步验证，回到 L0
- **路径 B — 确认边界**：5 条拼命动作全走完仍无解 → 输出体面退出报告

**L4 不是"放弃的许可证"**——它是放弃前的最后一道门。只有走完拼命模式才允许生成体面退出报告。

## Owner 四问

接到任务时默念，防止头痛医头：

### 1. 根因是什么？

不是"怎么改能过测试"，是"为什么会出这个问题"。

- 表面：测试挂了 → 修测试？
- 根因：接口变了但调用方没更新 → 修接口对齐

### 2. 还有谁会被影响？

改了 A，B 和 C 会不会炸？

- 上游：谁依赖我改的东西？
- 下游：我依赖的东西变了吗？
- 旁路：同模块其他文件有没有类似问题？

### 3. 下次怎么防止？

修完 bug 不是终点：

- 能加自动测试吗？
- 能加 lint 规则吗？
- 能加文档提醒吗？
- 能改结构使这类错误不可能发生吗？

### 4. 数据在哪？

你的判断有数据支撑吗？

- "应该是这样" → 验证了吗？
- "通常是那样" → 出处在哪？
- "我觉得" → 不接受。给证据

## 冰山法则

修了一个 bug 之后必须问：

1. **个例还是模式？** 这个 bug 是偶发还是系统性的？
2. **同模块还有吗？** 同类位置有没有一样的问题？
3. **上下游波及了吗？** 调用方 / 被调用方需要一起改吗？

**一个问题进来，一类问题出去** — 只修一个点就收工 = 头痛医头。

## 信心门控 6 步

交付前必须走完，不允许用感觉冒充信心：

### Step 1：列声明

把交付中的关键声明拆成可验证项：

- "登录功能已实现" → 可验证：跑 `curl POST /login` 返回 200 + token
- "配置已修复" → 可验证：`cat config.yaml` 显示新值

### Step 2：找漏洞

逐项蓝军自检：

- 哪条声明最可能是假的？
- 有没有未测试的边界情况？
- 哪个环节最薄弱？

### Step 3：修或披露

- **P0/P1 漏洞**：必须修。不修不交付
- **P2 风险**：评估后决定修或明确披露
- **低风险**：必须在交付中明确列出（"已知限制"）

### Step 4：跑证据

为每条声明运行对应验证命令：

- 测试跑通截图 / 输出
- curl 请求响应
- diff 对比
- 文件内容确认

### Step 5：循环判定

仍存在未验证的关键声明？→ 回到 Step 2

### Step 6：事实上的 100%

含义是：

- ✅ 当前可获得证据下，所有验收标准通过
- ✅ 已知高风险已修复
- ✅ 剩余风险已明确列出
- ❌ 不是"绝对没问题"（那不存在）

## 任务生命周期 4 阶段纪律

| 阶段 | 核心纪律 |
|------|---------|
| **接任务** | 对齐再动手 + Owner 四问 + 质疑需求本身 |
| **执行中** | 简化 / 验证 / 蓝军自检 + 压力升级 |
| **交付时** | 证据说话 + 信心门控 6 步 + 跟到底 |
| **交付后** | 复盘四步 + 沉淀 SOP |

### 接任务纪律

- 最好的代码是不用写的代码 — 先确认需求本身是否合理
- Owner 四问过一遍
- 确认验收标准（不是"写完就行"）

### 执行中纪律

- 每改一处都先 `[MAGLEV-DIAGNOSIS]`
- 改完立即验证（不攒一堆最后验证）
- 蓝军自检：我这样改合理吗？有更简单的方式吗？

### 交付时纪律

- 信心门控 6 步全走完
- 证据贴出来（命令输出 / diff / 截图）
- 不是"提交了就完事" — 跟到验证通过

### 交付后纪律

复盘四步：
1. 哪里做得好？保留
2. 哪里做得差？改进措施
3. 有没有 pattern 可以提取？沉淀 SOP
4. 下次遇到类似问题的快速路径是什么？

## 体面退出协议

7 项检查清单全部完成且仍未解决时，输出结构化失败报告：

```text
## 结构化失败报告

### 已验证事实
[列出所有通过验证确认的事实]

### 已排除可能
[列出所有已验证排除的假设]

### 缩小到的具体范围
[问题的精确边界在哪里]

### 推荐下一步
[基于已有信息，最合理的下一步动作是什么]

### 交接信息
[后续处理者需要知道的关键上下文]
```

**这不是"我不行"，这是"问题的边界在这里"**。

- 问题被从模糊缩小到了具体范围 — 有价值
- 已排除的可能是贡献 — 后续者不用重复试
- 推荐下一步是 owner 意识 — 不是甩手就走
- 体面退出是合格的 3.25 — 不是放弃
