# Reverse Principles

## 目标
定义 `maglev-reverse-spec` 在任意存量项目中的通用逆向原则，确保产物既符合 Maglev 规范，又不把推测误写成事实。

## 核心原则

### 1. Evidence First
- 先证据，后结论。
- 每个关键判断都应尽量附带代码、配置、日志、接口定义或测试作为锚点。
- 无法锚定的内容必须降级为 `[INFERENCE]` 或 `[UNKNOWN]`。

### 2. Structure Before Detail
- 优先还原模块骨架，再深入局部细节。
- 骨架至少包括：入口、核心流程、数据结构、文件结构、依赖关系。
- 避免一开始陷入局部实现，导致整体视角缺失。

### 3. Multi-Layer Reconstruction
- 逆向不是只看某一层代码。
- 必须尽量跨层理解：入口层、编排层、数据层、状态层、运行时层。
- 若系统存在前后端、事件流、配置流、缓存流，应显式标注层间映射。

### 4. Facts, Inferences, Unknowns
- 使用统一语义区分信息可信度：
- `[FACT]`：有明确证据支撑。
- `[INFERENCE]`：基于证据推导，但尚未被直接证明。
- `[UNKNOWN]`：当前证据不足，无法可靠判断。
- `[QUEST]`：需要进一步向专家、日志、线上行为或补充材料求证的问题。

### 5. Data Is First-Class
- 数据结构不是附属信息，而是逆向主线之一。
- 必须优先识别系统操作的核心对象、字段约束、跨层映射、生命周期与状态承载体。
- 当“流程”和“数据”冲突时，优先回到数据结构核对真实语义。

### 6. Runtime Matters
- 静态代码结构不足以解释真实系统行为。
- 在证据允许时，应补充运行时信息：异步链路、缓存、并发、重试、降级、失败路径。
- 对运行时部分缺乏证据时，要明确说明观察盲区。

### 7. Generic Method, Contextual Output
- 方法论必须通用，但输出应适配当前项目上下文。
- 允许保留 Maglev 的目录结构、交付格式和验证要求。
- 不应把某个具体业务项目的领域词汇、模块名称、表名、接口名固化为 skill 本体。

### 8. Actionable Deliverables
- reality 产物不仅要“看懂”，还要能支持后续动作。
- 至少要能帮助用户完成：交接、重构前分析、契约核对、风险识别、问题求证。
- 若产物无法指导下一步工程动作，则说明逆向深度不足。

### 9. Calibrated Depth
- 根据目标选择 `Lean / Standard / Deep` 输出档位。
- 不为完整而完整，不为简化而遗漏关键骨架。
- 对高复杂、高风险模块，宁可冗余，也不要遗漏关键运行时与风险信息。

### 10. Traceable Handoff
- 最终产物必须能追溯回原始实现。
- 关键章节应附 `Implementation Trace`、文件路径、函数名、接口名或结构定义位置。
- 逆向结果必须支持后续审计，而不只是阅读。

### 11. Reverse Is Observational, Not Remedial
- 逆向阶段的职责是发现 reality，不是改变 reality。
- 默认禁止在 reverse 过程中直接修业务代码、补回填工具、执行数据修复、变更接口契约或优化运行时逻辑。
- 逆向可以记录问题、量化风险、提出建议，但不能因为“已经发现问题”就自动进入修复。
- 若用户明确要求修复，必须把任务重新定性为实现/治理任务，并与 reverse 产物分开处理。

### 12. Chinese-First Deliverables
- reverse / reality 产物默认以中文为主。
- 允许英文保留在代码标识符、接口路径、字段名、日志原文和第三方专有名词中。
- 除非用户明确要求英文或其他语言为主，否则章节标题、正文说明、结论、风险、建议、评分和问题队列都必须使用中文。
- 若引用英文原文，应以“原文保留 + 中文解释”的方式呈现，而不是让整份产物滑向英文。
