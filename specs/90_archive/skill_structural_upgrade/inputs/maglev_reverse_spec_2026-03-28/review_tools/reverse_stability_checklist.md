# Reverse Stability Checklist

> 用途：用于判断一份逆向结果是否已经达到“稳定可用”，而不只是“看起来写得很多”。

## 1. 稳定性的定义

这里的“稳定”指的是：

- 逆向结果在不同入口下复核时，不会大幅改写主结论
- 结果足够支撑后续交接、改动分析或继续实现
- 未知项被集中管理，而不是混入已确认结论

稳定不等于“没有未知项”。
稳定的关键在于：

- 已知部分足够扎实
- 推断边界足够清楚
- 未知部分足够收口

## 2. 快速结论档位

- `S1 Unstable`
  - 骨架不完整
  - 事实和推断混杂
  - 换入口复核后主结论容易变
- `S2 Usable`
  - 骨架基本完整
  - 结果可用于理解、交接和局部改动分析
  - 仍有若干关键未知
- `S3 Stable`
  - 骨架完整
  - 跨层一致
  - 未知项集中收口
  - 复核后主结论基本不变

## 3. 第一层检查：骨架完整性

至少应具备以下内容：

- Feature Map
- Evidence Summary
- Data Structure Map
- Data Dictionary
- Architecture Overview
- Main Flow
- File Structure
- Implementation Trace
- Unknowns / Quests

### 判断规则
- 缺 3 项及以上：直接判 `S1`
- 缺 1-2 项：最多判 `S2`
- 全部具备：可进入下一层判断

## 4. 第二层检查：证据边界

检查文档里是否明确区分：

- `[FACT]`
- `[INFERENCE]`
- `[UNKNOWN]`
- `[QUEST]`

### 重点看什么
- 关键结论是否有代码、测试、接口或运行痕迹支持
- 推断是否被显式降级
- 未知项是否被单独列出

### 红线
若存在以下情况，直接降级：
- 把推测写成确定需求
- 大量描述没有证据支撑
- 文档中“应该/大概/可能”泛滥，但未做类型标记

## 5. 第三层检查：跨层一致性

这是稳定性判断里最关键的一层。

需要核对：

- 前端字段 与 API 契约 是否一致
- API 契约 与 后端结构 是否一致
- 后端结构 与 数据字典 是否一致
- 状态变化 与 主流程描述 是否一致
- 事件载荷 / 缓存对象 / Store State 是否与主对象语义一致

### 常见不稳定信号
- 字段同名异义
- 前后端分页参数不一致
- 响应结构和类型定义不一致
- 文档把状态机画出来了，但代码里找不到对应状态
- 文档说“有鉴权”，实现里其实没看到

若跨层漂移较多，即使文档很多，也不稳定。

## 6. 第四层检查：未知项收口

好的 reverse 结果不会假装什么都知道，而是会把未知项控制在可管理范围内。

检查：

- Unknowns 是否集中列出
- Quest 是否可行动
- 是否区分了“当前不知道”和“需要专家确认”
- 未知项是否影响主结论稳定性

### 判断规则
- 未知项散落全文：偏不稳定
- 未知项集中在 `Unknowns / Expert Review Queue`：偏稳定
- 若未知项正好落在主链路核心位置：最多判 `S2`

## 7. 第五层检查：换入口复核

这是最强的稳定性验证。

### 做法
从另一个入口重新逆一次关键主线，例如：

- 先从 UI 跑过，再从 API 入口复核
- 先从 API 跑过，再从数据结构入口复核
- 先从主流程跑过，再从状态机入口复核

### 稳定结果应表现为
- 核心对象一致
- 主流程一致
- 核心状态一致
- 关键风险点一致

### 不稳定结果常见表现
- 第二次复核后，连“模块主要做什么”都变了
- 数据主对象改名换义
- 主流程顺序被完全推翻
- 原本的关键状态机不成立

## 8. 第六层检查：工程可用性

问这 3 个问题：

1. 后续接手人能否直接基于它继续做事？
2. 如果要改这个模块，风险点是否已经暴露？
3. 如果要补测试或补设计，它能否直接提供输入？

### 判断规则
- 3 个都答不上来：`S1`
- 至少 2 个可以明确回答：`S2`
- 3 个都能明确回答：可考虑 `S3`

## 9. 评分表

总分 20 分：

- 17-20: `S3 Stable`
- 13-16: `S2 Usable`
- 0-12: `S1 Unstable`

| 项目 | 分值 | 判断标准 |
|---|---:|---|
| 骨架完整性 | 4 | 图、流程、结构、trace 是否齐 |
| 证据边界清晰 | 4 | FACT / INFERENCE / UNKNOWN / QUEST 是否明确 |
| 跨层一致性 | 4 | 前后端、数据、状态、契约是否对得上 |
| 未知项收口 | 3 | Unknowns / Quests 是否集中、是否可行动 |
| 换入口复核稳定 | 3 | 二次复核后主结论是否基本不变 |
| 工程可用性 | 2 | 能否支持交接、改动分析、继续实现 |

## 10. 最小验收问题

如果你不想打完整分，可以只问这 5 个问题：

1. 这份 reverse 有没有完整骨架？
2. 哪些是事实，哪些是推断，哪些还不知道，能不能一眼看出来？
3. 数据结构、接口契约、状态流能不能互相对上？
4. 未知项是不是被集中管理了？
5. 换一个入口再看，主结论会不会大变？

只要有 2 个问题答不上来，这份结果通常还不稳定。

## 11. 评审输出模板

```markdown
# Reverse Stability Review

## 结论
- Rating: S1 / S2 / S3
- Score: {score}/20
- Summary: {一句话总结}

## 主要发现
- [F1] ...
- [F2] ...

## 不稳定点
- [R1] ...
- [R2] ...

## 建议动作
1. ...
2. ...
3. ...
```

## 12. 推荐使用方式

建议在另一个项目里这样用：

1. 跑完第一版 reverse
2. 用这份清单做一次人工评审
3. 从另一个入口复核主链路
4. 再做一次稳定性结论

如果两轮评审后主结论基本没变，这份 reverse 结果就可以认为已经比较稳定。
