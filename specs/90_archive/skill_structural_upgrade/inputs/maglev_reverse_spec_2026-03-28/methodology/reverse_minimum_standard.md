# 逆向产物最低合格标准

## 目标
定义一份“逆向结果至少该长什么样”的最小标准，用来判断一份 reverse 产物是否足够支持理解、交接或后续工程动作。

## 一、最低合格骨架

一份合格的逆向产物，至少必须包含以下 6 类内容：

1. 功能地图
2. 数据结构图或数据字典
3. 架构图
4. 主流程图
5. 文件结构
6. 实现追踪

如果缺任意一类，这份逆向结果通常都只能算“阅读笔记”，还不能算真正可交接的 reverse 产物。

## 二、章节最低要求

### 1. Index
至少回答：
- 这个模块叫什么
- 当前逆向范围是什么
- 入口类型是什么
- 当前输出档位是什么
- 关联文档有哪些

### 2. Requirements
至少回答：
- 这个模块解决什么问题
- 主要用户是谁
- 核心输入输出是什么
- 哪些结论是事实，哪些是推断，哪些未知
- 至少一版 User Story / AC

### 3. Design
至少回答：
- 系统由哪些关键组件构成
- 请求或事件从哪里进，从哪里出
- 核心对象有哪些
- 主流程怎么流转
- 哪些文件承载了这些逻辑

## 三、图形最低要求

### 1. 架构图
必须有。

最低标准：
- 标出前端/后端/服务/存储等关键节点
- 标出主调用方向

不要求一开始就非常精细，但不能完全没有。

### 2. 主流程图
必须有。

最低标准：
- 至少覆盖一个主链路
- 能看出触发点、核心处理、结果返回

如果模块有多个高风险动作，`Deep` 档应补多条流程图。

### 3. 文件结构图
必须有。

最低标准：
- 列出这次逆向真正纳入范围的关键文件
- 让读者知道“应该去哪里继续读”

## 四、数据最低要求

### 1. Core Data Structures
必须有。

最低标准：
- 名称
- 类型
- 定义位置
- 用途

### 2. Data Dictionary
`Standard` 起建议必须有。

最低标准：
- 关键字段
- 类型
- 是否必填
- 字段含义

### 3. Cross-Layer Mapping
只要存在前后端/接口/存储映射，就应该有。

最低标准：
- 请求结构 -> 服务对象 -> 持久化对象 -> 响应结构

## 五、追踪最低要求

### 1. Evidence Summary
必须有。

最低标准：
- `[FACT]`
- `[INFERENCE]`
- `[UNKNOWN]`

### 2. Implementation Trace
必须有。

最低标准：
- 每个关键结论至少能追到文件
- 最好能追到模块或函数

如果完全没有 trace，这份逆向结果不能用于交接。

## 六、档位分界线

### Lean 的最低边界
能让读者回答：
- 这是什么模块
- 从哪进
- 主流程是什么
- 核心数据是什么
- 还有哪些没搞清楚

### Standard 的最低边界
在 Lean 基础上，能让读者继续做：
- 文档交接
- 缺陷排查
- 中等规模改动前分析

因此 `Standard` 至少应补：
- Data Dictionary
- State Machine 或等价状态描述
- Change Risk
- Test Mapping

### Deep 的最低边界
在 Standard 基础上，能支撑：
- 高风险重构
- 架构审计
- 关键模块设计讨论

因此 `Deep` 至少应补：
- Runtime Behavior
- Security Surface
- Error Taxonomy
- RMM Scorecard
- Expert Review Queue

## 七、常见不合格表现

以下情况通常意味着逆向结果还不合格：

1. 只有文字总结，没有图
2. 只有 API 列表，没有数据结构
3. 只有代码路径，没有流程
4. 只有流程，没有文件范围
5. 只有结论，没有 Evidence / Trace
6. 没有 Unknowns，像是“全都懂了”，但实际上只是脑补

## 八、推荐验收问题

验收一份逆向结果时，可以快速问这 8 个问题：

1. 我能一眼看懂这是哪个模块吗？
2. 我知道它的入口在哪吗？
3. 我知道主流程怎么走吗？
4. 我知道核心数据对象和字段吗？
5. 我知道关键文件在哪吗？
6. 我分得清哪些是事实、哪些是推断吗？
7. 我知道当前还有哪些未知点吗？
8. 如果让我明天改这个模块，我知道风险大概在哪吗？

如果其中 3 个以上回答不了，这份逆向结果就需要继续补。
