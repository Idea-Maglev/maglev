## 2. 从仓库现状提炼出的主流程草案

当前讨论中提炼出的主流程，不建议继续简化为：

`Issue -> Spec -> Code`

而更接近：

`Reality -> Issue -> Intent -> Spec -> Execute -> Validate -> Crystallize`

## 3. 阶段定义草案

### 3.1 Reality

**目标**：
先理解当前项目真实处于什么状态，而不是直接进入需求讨论。

**当前判断**：

- `Reality` 是主流程起点，不是附属背景步骤
- 任何新 issue 都应建立在“当前现实是什么”的基础上

### 3.2 Issue

**目标**：
显式化一个需要处理的问题、机会、缺口或冲突。

**当前判断**：

- `Issue` 只是入口容器，不等于已经澄清的需求
- 它负责提出“要处理什么”，但不应直接承担完整设计职责

### 3.3 Intent

**目标**：
把 issue 从入口问题收敛成可进入设计的清晰意图。

**当前判断**：

- 这是当前体系里最值得单独定义的一段
- 许多低质量或妥协式 skill，很可能正是因为这一段没有被正式定义，才被迫长出来

**这一段当前至少需要回答**：

- 这次真正要改变什么
- 本次范围是什么，不是什么
- 成功标准大致是什么
- 它与当前 Reality 的冲突点是什么

### 3.4 Spec

**目标**：
把已经收敛的意图转成可执行设计依据。

**当前判断**：

- `Spec` 不应继续承担过多“需求猜测”和“问题定义”职责
- `Spec` 更应聚焦于设计、模块边界、流程、接口、约束和验收映射

### 3.5 Execute

**目标**：
基于 Spec 落地改动。

**当前判断**：

- `Execute` 应是清晰的实施阶段
- 但它不应吞掉完整验证职责

### 3.6 Validate

**目标**：
确认结果是否真正符合 Intent 与 Spec，而不只是“代码已写完”。

**当前判断**：

- 这是质量门，不只是实施附带动作
- 它应回答“是否真的解决了最初问题”

### 3.7 Crystallize

**目标**：
将本次执行后真正成立的新事实写回体系。

**当前判断**：

- `Crystallize` 不等于简单归档
- 它至少包含：
  - Reality 更新
  - 过程资产归档
  - 索引或结构认知更新

## 4. 当前识别出的关键错位

### 4.1 Issue 与 Intent 未稳定分离

当前很可能存在从 issue 直接滑向 spec 的问题。

### 4.2 Reality 与 Evolution 理念上清楚，执行上仍易混淆

当前风险包括：

- 把未来计划误写成当前现实
- 把过程性施工内容过早结晶成 Reality

### 4.3 Spec 承担了过多角色

当前 `Spec` 很可能同时承担：

- 问题定义
- 方案设计
- 部分计划
- 验收映射

### 4.4 Execute 与 Validate 的边界不稳定

当前存在多种实施和质量 skill 并存，说明执行与验证接口尚未完全稳定。

### 4.5 Crystallize、Archival、Closeout 概念可能混在一起

如果不拆清，容易出现：

- 任务结束了，但 Reality 没更新
- Reality 更新了，但过程资产没归档
- 归档做了，但并没有新事实成立

## 6. `Issue`、`Intent`、`Evolution` 关系对齐草案

### 6.1 当前问题

当前实际操作中，最容易混淆的是三件事：

- `issue` 被当成了进行中的需求正文
- `Spec` 被迫承担问题定义与意图澄清
- `specs/20_evolution/active/` 没有被稳定用作“施工中的需求承载层”

这会导致两类偏差：

1. 讨论和正文散落在 `issues/`，而 `evolution` 为空
2. 一旦进入 `evolution`，又直接跳到方案或实现，缺少中间意图层

### 6.2 第一版关系判断

当前建议将三者关系收敛为：

#### A. `issue`

`issue` 的职责应是：

- 记录触发来源
- 说明为什么值得处理
- 提供追踪 ID、外部上下文和任务入口

它更像：

> trigger / ticket / handle

而不是进行中的需求正文。

换句话说：

- `issue` 应该告诉我们“为什么要启动这件事”
- 但不应该承载完整的对齐过程和演进内容

#### B. `intent`

`intent` 的职责应是：

- 将 `issue` 从入口问题收敛为清晰改动目标
- 说明本轮真正要改变什么
- 划清本次范围与非目标
- 给出进入设计前的最小共识

它更像：

> clarified change target

也就是说：

- `issue` 解决“为什么开始”
- `intent` 解决“这次到底要改什么”

#### C. `evolution`

`specs/20_evolution/active/` 的职责应是：

- 承载进行中的需求正文
- 承载上下文对齐过程
- 承载设计、验证前提和迭代中的修正

它更像：

> active construction site

也就是说：

- 一旦需求进入真实推进状态
- 主体内容就应落在 `evolution`
- 而不是继续悬挂在 `issues/`

### 6.3 当前建议的顺序关系

第一版建议顺序为：

`Issue -> Intent Clarification -> Evolution Active Spec -> Execution -> Crystallization`

其中：

- `Issue`
  - 负责启动
- `Intent Clarification`
  - 负责收敛目标
- `Evolution Active Spec`
  - 负责承载进行中的正文与演进过程

### 6.4 对当前仓库操作方式的直接影响

如果采用这套关系，后续操作应满足：

1. 新需求启动时，可以先存在 `issue`
2. 一旦确认要正式推进，应尽快在 `specs/20_evolution/active/<topic>/` 建立正文
3. 讨论过程、关键判断、阶段定义、开放问题应优先沉淀在 active spec 中
4. `issues/` 不再承担“进行中正文承载层”的职责

### 6.5 当前仍需继续讨论的点

尽管上述关系比当前状态更清楚，但仍有几个点需要继续对齐：

1. `intent` 是否需要独立物理文件
2. 如果需要，是否应位于 `specs/20_evolution/active/<topic>/00_context.md` 中，还是单独成为 `00_intent.md`
3. `issue` 与 active spec 之间是否需要强制 traceability 字段
4. 当任务很小且高度原子时，是否允许 `issue` 与 `intent` 合并表达

## 7. 当前倾向性判断

截至本轮讨论，当前更倾向于以下判断：

1. `issue` 不应承载进行中的需求正文
2. `specs/20_evolution/active/` 才是进行中需求的主承载层
3. `intent` 是真实存在的阶段，只是尚未被稳定产品化
4. 当前很多技能结构问题，很可能都和 `intent` 没有被单独稳定承载有关

## 8. `intent` 物理承载方式候选

### 8.1 当前需要回答的问题

如果 `intent` 是真实存在的阶段，那么接下来要决定的不是“它是否重要”，而是：

> 它在 `specs/20_evolution/active/<topic>/` 中应该如何被物理承载？

当前至少有三种候选方案。

### 8.2 方案 A：`intent` 继续隐含在 `01_requirements.md`

#### 结构

```text
specs/20_evolution/active/<topic>/
├── 00_context.md
└── 01_requirements.md
```

其中：

- `00_context.md` 记录讨论背景、开放问题、推翻记录
- `01_requirements.md` 同时承载：
  - issue 收敛结果
  - intent
  - requirements

#### 优点

- 文件数最少
- 对当前仓库侵入最小
- 容易延续现有 `01_requirements.md` 习惯

#### 风险

- `intent` 很容易再次被 requirements 吞掉
- “问题定义”和“需求定义”混在一个文件里，后期仍会膨胀
- 很难明确区分“为什么改”和“具体要满足什么”

#### 当前判断

这是最保守的迁移方式，但很可能只是延续当前模糊状态，不足以真正解决边界问题。

### 8.3 方案 B：新增独立 `00_intent.md`

#### 结构

```text
specs/20_evolution/active/<topic>/
├── 00_context.md
├── 00_intent.md
└── 01_requirements.md
```

其中：

- `00_context.md`：记录讨论过程、分歧、阶段判断、被修正结论
- `00_intent.md`：记录本轮改动意图、本次范围、非目标、成功标准、与 Reality 的冲突点
- `01_requirements.md`：记录正式需求内容

#### 优点

- `intent` 边界最清楚
- 能明确区分：
  - 为什么改
  - 这次到底要改什么
  - 要满足什么要求
- 有利于后续让 `create-spec` 只消费已经收敛的目标

#### 风险

- 文件数量增加
- 如果使用纪律不强，可能出现 `context` / `intent` / `requirements` 三者重复
- 需要明确三个文件的边界，否则只是把混乱拆成多个文件

#### 当前判断

这是最结构化、也最符合当前讨论方向的方案，但需要更严格的文件职责定义。

### 8.4 方案 C：将 `intent` 作为 `00_context.md` 的固定章节

#### 结构

```text
specs/20_evolution/active/<topic>/
├── 00_context.md
└── 01_requirements.md
```

但约定 `00_context.md` 必须包含固定章节：

- 背景
- 现状判断
- intent
- 开放问题
- 决策修订记录

#### 优点

- 不新增文件
- 比方案 A 更容易明确 `intent` 的存在
- 适合在讨论密集阶段快速推进

#### 风险

- `context` 文件会变得越来越重
- `intent` 仍可能被大量讨论内容淹没
- 当话题复杂时，不如独立文件清楚

#### 当前判断

这是一个折中方案，适合过渡期，但长期可能仍不够稳定。

## 9. 当前倾向

截至目前，当前更倾向于：

### 倾向结论

优先考虑 **方案 B：新增独立 `00_intent.md`**。

### 原因

1. 当前真正缺的不是“requirements 文件”，而是“进入 requirements 和 spec 之前的目标收敛层”
2. 如果 `intent` 继续隐含，很容易再次被 requirements 或 context 吞没
3. 只有把 `intent` 单独物理化，后面我们在讨论 skill 体系时，才能真正判断哪些 skill 在替意图澄清补洞

### 但仍需确认的风险

在采用方案 B 之前，仍需进一步对齐：

1. `00_context.md` 与 `00_intent.md` 的边界
2. `00_intent.md` 与 `01_requirements.md` 的边界
3. 是否所有 active spec 都需要 `00_intent.md`
4. 对非常小的需求，是否允许跳过独立 intent 文件

## 11. 独立 `intent` 的后续作用

### 11.1 需要先避免的误区

如果 `intent` 独立出来，但后续没有明确用途，它就会退化成：

- requirements 的前言
- context 的摘要副本
- 一份只在开头写一次、之后没人再读的文件

那它就不值得独立。

因此，独立 `intent` 必须在后续流程里承担持续作用，而不是一次性说明。

### 11.2 当前识别出的核心作用

#### A. 作为进入 `requirements` 和 `spec` 前的稳定边界

`intent` 的第一作用不是提供更多细节，而是提供边界。

它应该稳定回答：

- 这次真正要改变什么
- 本次不打算改变什么
- 为什么这次值得做
- 当前成功标准大致是什么

后续在写 `01_requirements.md` 和设计方案时，应默认不能随意突破这层边界。

也就是说：

> `intent` 是 requirements 和 spec 的约束面，不是它们的附属说明。

#### B. 作为需求漂移的判断基线

当前很多任务在推进过程中会发生需求漂移：

- 讨论越聊越大
- requirements 越写越多
- spec 开始吞入原本不在本轮范围内的内容

如果没有独立 `intent`，就很难判断这是正常细化，还是范围漂移。

所以 `intent` 的第二作用是：

> 当 requirements / spec / execution 发生扩张时，用来判断“这还是不是原本那件事”。

#### C. 作为验证阶段的目标基线

验证阶段不应该只回答“实现是否符合 spec”，还要回答：

> 这次改动是否真的解决了最初要解决的问题。

这里 `intent` 的作用是：

- 给 `Validate` 提供最初目标基线
- 防止验证层只检查设计一致性，却不检查目标一致性

换句话说：

- `spec` 更偏“怎么做”
- `intent` 更偏“为什么做、做到什么算对”

#### D. 作为后续结晶时的取舍依据

在 `Crystallize` 阶段，最大的难点之一是：

- 哪些变化属于真正成立的新事实
- 哪些只是本轮施工中的中间决策

独立 `intent` 在这里的作用是：

> 帮助判断本轮进入 Reality 的，到底应该是哪些“围绕本次目标真正成立的变化”。

也就是说，它不仅服务启动，还服务收口。

#### E. 作为技能治理的分析入口

对这次 `skill结构性升级` 讨论来说，`intent` 还有一个额外价值：

如果它独立存在，我们就能更清楚地识别：

- 哪些 skill 在服务“问题定义/目标收敛”
- 哪些 skill 在服务“requirements/spec 生产”
- 哪些 skill 在服务“执行/验证/结晶”

这会直接帮助后续判断：

- 哪些 skill 是真实阶段能力
- 哪些只是因为 `intent` 缺位而长出来的补丁

### 11.3 当前结论

如果 `intent` 要独立，它至少必须同时服务四个后续阶段：

1. 约束 `requirements`
2. 约束 `spec`
3. 作为 `validate` 的目标基线
4. 作为 `crystallize` 的取舍依据

只有这样，它才不是冗余文件，而是主流程中的稳定接口层。

### 11.4 由此带来的结构要求

这也意味着，独立 `intent` 不能写成泛泛的背景介绍，而必须具备：

- 明确的改动目标
- 明确的范围边界
- 明确的非目标
- 明确的成功判断

否则它无法承担上述作用。

后续继续讨论时，优先把新增结论写入本文件，并持续维护以下三类内容：

- 新共识
- 新开放问题

## 13. `00_intent.md` 最小模板草案

### 13.1 设计目标

如果 `00_intent.md` 要成立，它的模板必须满足两个条件：

1. 足够小
   - 不要再次长成 requirements 或 spec
2. 足够硬
   - 能作为后续 requirements / spec / validate / crystallize 的共同约束面

因此它不应该追求“写全”，而应该追求“固定边界”。

### 13.2 最小字段建议

当前第一版建议，`00_intent.md` 至少包含以下字段。

#### A. `Change Target`

**回答的问题**：

- 这次真正要改变什么

**要求**：

- 用 1-3 句写清本轮改动目标
- 必须指向“变化对象”，而不是空泛愿景

**作用**：

- 作为后续 requirements 和 spec 的总目标锚点

#### B. `Why Now`

**回答的问题**：

- 为什么这件事现在值得启动

**要求**：

- 写清当前触发原因
- 可以是结构风险、能力缺口、质量问题、现实错位、演进阻塞

**作用**：

- 防止任务启动理由在后续推进中丢失
- 帮助判断后续扩张是否仍围绕原始问题

#### C. `In Scope`

**回答的问题**：

- 本次明确要处理什么

**要求**：

- 使用短列表列出本轮承诺处理的范围
- 应该能映射到后续 requirements

**作用**：

- 约束需求膨胀
- 为后续拆 requirements 提供边界

#### D. `Out of Scope`

**回答的问题**：

- 本次明确不处理什么

**要求**：

- 写明最容易被顺手卷入、但本轮不打算处理的内容

**作用**：

- 防止 requirements / spec / execution 阶段自然扩张
- 在讨论中形成明确拒绝面

#### E. `Success Signal`

**回答的问题**：

- 到什么程度可以认为这次目标已经成立

**要求**：

- 不写成详细验收清单
- 只写 2-4 条高层成功信号

**作用**：

- 作为 Validate 阶段的目标基线
- 帮助区分“实现完成”和“问题解决”

#### F. `Reality Tension`

**回答的问题**：

- 当前现实中，是什么让这次变化变得必要

**要求**：

- 明确指出与现状的张力
- 可以是当前 skill 结构、当前流程、当前目录语义、当前运行方式等

**作用**：

- 把 intent 和 Reality 接上
- 帮助 Crystallize 阶段判断最终哪些变化应回写 Reality

### 13.3 当前不建议放进 `intent` 的内容

以下内容当前不建议直接放进 `00_intent.md`：

- 详细功能需求
- 设计方案
- 接口/数据结构
- 实施计划
- 详细测试用例
- 长篇背景介绍

这些内容更适合分别进入：

- `01_requirements.md`
- 后续 design / plan / test 产物
- `00_context.md`

### 13.4 第一版模板骨架

当前建议骨架如下：

```md
# <topic> Intent

## Change Target

## Why Now

## In Scope

## Out of Scope

## Success Signal

## Reality Tension
```

### 13.5 当前判断

这套模板的重点不是丰富表达，而是提供一个足够稳定的“流程接口文件”。

如果后续验证发现它仍然太空泛，可以再补字段；  
但当前不宜一开始就把它做成第二份 requirements。

## 14. 基于当前主题的 intent 试填样例

为了验证模板是否有信息增量，下面先用当前 `skill结构性升级` 主题做一版试填。

### Change Target

将 Maglev 当前分散、边界模糊的 skill 体系，收敛为一套与主流程一致的能力结构；优先解决 issue、intent、evolution、reality 之间的流程错位，以及由此带来的 skill 膨胀问题。

### Why Now

当前 skill 数量已经增长到需要结构治理的阶段；如果继续在未对齐主流程的前提下增量修 skill，只会放大后续维护复杂度，并继续制造补丁式能力。

### In Scope

- 重新识别 Maglev 当前主流程
- 对齐 `issue`、`intent`、`evolution`、`reality` 的关系
- 定义 `intent` 是否需要独立产物层
- 反推 skill 体系应该如何围绕主流程重组

### Out of Scope

- 立即重写全部 skill
- 逐个 skill 的内容优化
- 直接启动公开分发链路重构
- 在未完成流程对齐前先做全面 skill 裁撤

### Success Signal

- 能清楚说明进行中需求应承载在 `issues/` 还是 `specs/20_evolution/active/`
- 能清楚说明 `intent` 在主流程里的职责和后续作用
- 能基于对齐后的流程重新判断 skill 分层，而不是继续按现有目录并列理解

### Reality Tension

- 当前 `specs/20_evolution/active/` 为空，说明进行中需求没有稳定进入 Evolution 层
- 当前 issue 与 active spec 的实际分工不清
- 当前部分 skill 很可能是在流程接口缺失的情况下被迫长出的补丁

### 13.6 当前开放问题

1. `Reality Tension` 是否应该保留这个名字，还是换成更直接的 `Current Tension`
2. `Success Signal` 是否足够，还是需要补一项 `Failure Signal`
3. 对于极小需求，是否允许把 `Why Now` 与 `Reality Tension` 合并

## 15. 来自 `inputs/maglev_reverse_spec_2026-03-28` 的新增输入

### 15.1 当前结论

本次在其它项目中针对 `maglev-reverse-spec` 的单点优化，对当前 `skill结构性升级` 主线是有帮助的。

帮助点不主要在“reverse skill 本身怎么改”，而在于它已经把以下问题做成了可操作标准：

- `reality` 应该细到什么程度才有工程价值
- 一份产物怎样才算稳定、可交接，而不是只是“看起来写得很多”
- 一种 skill 怎样保持通用性，而不是被单项目经验污染

这三点，都会直接反作用于当前 skill 体系升级。

### 15.2 对当前主线最有价值的输入

#### A. `reality` 不能只是摘要，必须达到“工程可用”

在 `methodology/reverse_minimum_standard.md` 中，已经把一个合格 reverse/reality 产物的最低骨架写得很清楚：

- 功能地图
- 数据结构
- 架构图
- 主流程图
- 文件结构
- 实现追踪

这对当前主线的启发是：

> `Reality` 的价值不在“解释得像结论”，而在“能不能支撑后续理解、交接和工程动作”。

也就是说，后续我们讨论 `Reality` 在主流程中的位置时，不能只把它当作一页运营摘要或状态总览。

#### B. `reality` 需要稳定性标准，而不是只看详细程度

在 `review_tools/reverse_stability_checklist.md` 和 `reverse_stability_quickcheck.md` 中，已经给出了更关键的判断：

- 稳定不等于没有未知项
- 稳定意味着：
  - 骨架完整
  - 事实/推断/未知边界清楚
  - 跨层一致
  - 未知项集中收口
  - 换入口复核后主结论不大变

这对当前主线的启发是：

> 后续 Maglev 在谈 `Reality` 时，不应只谈“要写多细”，还必须谈“什么样的 Reality 才算稳定可用”。

这会直接影响：

- `standup` 应该依赖什么级别的事实
- `intent` 与 `crystallize` 之间如何判断哪些内容可以写回 `10_reality`
- 哪些 skill 应负责把施工信息提纯为稳定事实

#### C. `Reality` 的粒度需要按模块分区，而不是混写

在 handoff checklist 和 diff summary 中，有一条非常关键的约束：

- `Module Partition`
- reality 按 `module_slug` 单独落盘
- 禁止多模块混写

这对当前主线的启发是：

> `Reality` 不是越大越好，而应该按可管理粒度分区。

这和当前 `skill结构性升级` 讨论里的“目录即架构”问题是同一个方向：

- 如果 Reality 粒度不稳，后续 skill 结构也容易继续膨胀或混写

#### D. “通用性检查”本身可以迁移到 skill 体系治理

在 `reverse_skill_genericity_checklist.md` 中，已经把一个 skill 是否保持“Maglev-compatible but business-neutral”做成了明确清单。

这对当前主线的启发是：

> 后续 skill 结构升级不应只讨论分层，还应补一套“通用性与污染检查”机制。

也就是说，未来我们可能不仅需要：

- 生命周期元数据
- 层级元数据

还需要一类面向治理的检查维度：

- 是否项目污染
- 是否方法论通用
- 是否只是示例偏置

#### E. 详细程度应该和输出档位绑定，而不是一刀切

在逆向优化中，`Lean / Standard / Deep` 这套输出档位已经被收敛出来。

这对当前主线的启发是：

> 对 `Reality` 和 skill 产物的详细程度，也许不该只用单一标准，而应允许存在不同档位。

这会帮助我们避免另一个极端：

- 为了“工程可用”，把所有 skill / reality 都写得过重

### 15.3 对当前需求的直接影响

基于这批输入，当前 `skill结构性升级` 主线需要新增三个约束：

1. 后续讨论 `Reality` 时，必须同时讨论：
   - 最低工程可用骨架
   - 稳定性标准
   - 粒度分区原则

2. 后续讨论 skill 元数据时，除了层级与生命周期，还应考虑：
   - 通用性 / 污染检查维度

3. 后续若要定义新的结构标准，应避免“一刀切详细”，而应考虑：
   - 档位化输出

### 15.4 当前判断

这批逆向优化输入不会替代当前主线，但它对当前主线有明显帮助，尤其是在下面三个方面：

1. 它让 `Reality` 从抽象概念变成了可判定的工程产物
2. 它提示我们：skill 结构治理不能只看分层，还要看稳定性与通用性
3. 它说明“详细程度”本身也应被结构化，而不是靠经验拍脑袋决定

## 16. 新增约束：减少抽象比喻驱动的 skill 设计

### 16.1 问题来源

当前部分旧 skill 的生成方式，存在明显的“抽象化、人设化、比喻驱动”倾向。

典型例子是：

- 用角色人格替代能力边界
- 用比喻性命名替代结构定义
- 用叙事氛围替代输入/输出与职责说明

这类写法在早期可能提升了表达趣味，但也带来了明显副作用：

- 解释成本上升
- 边界不清
- 真实能力与文案形象脱节
- 后续维护时很难判断哪些内容是协议，哪些只是修辞

### 16.2 当前判断

后续 skill 生成与重构时，应减少这类抽象主观比喻的使用，尤其不能再让它主导 skill 的结构表达。

更明确地说：

> 比喻可以辅助阅读，但不能承担定义能力边界的职责。

### 16.3 对后续 skill 设计的直接约束

后续 skill 设计应优先写清：

- 这个 skill 解决什么问题
- 它处于系统哪一层
- 它的输入是什么
- 它的输出是什么
- 它与相邻能力如何衔接

而不应优先写：

- 它像什么角色
- 它是什么人格
- 它代表什么世界观隐喻

### 16.4 对 `atomizer` 的特别启发

`atomizer` 当前的问题不只是“是不是入口层”，还包括它采用了较强的人格化与抽象化表达。

这使得它在结构上更难被重判，因为：

- 它既像路由器
- 又像会话助手
- 又像解释层
- 但这些职责没有通过严格结构定义被清楚拆开

这进一步说明：

后续不仅要重判 `atomizer` 的层级，还要重判这类 skill 的表达方式。

## 17. 新增对齐点：`quick-dev` 到底是什么能力

### 17.1 当前问题

随着 `atomizer`、`spec pipeline`、`quality surfaces` 被逐步收清，`quick-dev` 的结构定位也开始变得需要重新确认。

当前需要明确的不是：

- `quick-dev` 能不能写代码

而是：

> `quick-dev` 在新结构里到底承接的是哪一种能力？

### 17.2 当前可能存在的歧义

`quick-dev` 这个名字天然会带来几种不同理解：

1. 它是“快速开发模式”
2. 它是“默认实施入口”
3. 它是“小范围改动的施工能力”
4. 它是“在 spec 之后直接落地实现的主执行对象”

这些理解彼此相近，但结构上并不完全一样。

### 17.3 当前需要继续回答的问题

1. `quick-dev` 是否等于 `Execute` 阶段本体
2. 它是否只适合“小范围、边界清晰”的需求
3. 对复杂需求，是否应该存在比 `quick-dev` 更上层或更完整的实施能力
4. 它和质量层、spec 层的边界应如何写清

### 17.4 当前判断

`quick-dev` 很可能不是“所有实施能力的总名”，而更像：

> 面向已收敛需求的小步实施对象

如果这一判断成立，后续对 `Core Flow Layer` 的说明就还需要继续收细。

### 17.5 新增校准

当前进一步判断：

`quick-dev` 这个命名本身存在方向性错误。

问题不只是它是否覆盖全部实施能力，而是：

- 它把能力重点错误地放在“quick”
- 容易把实施理解成“追求快”
- 而不是“基于上下文与执行依据完成编码任务”

因此，后续对这一对象的重构，不能只做边界澄清，还应同步重做命名。
