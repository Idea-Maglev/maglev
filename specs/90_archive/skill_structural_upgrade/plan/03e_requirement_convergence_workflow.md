# skill结构性升级 需求收敛 Workflow 草案

> 状态：草案
> 目标：为当前缺位的 `需求收敛` 能力提供最小可执行 workflow 结构。

## 1. 定位

`需求收敛` 当前是未来 Maglev 用户显性一级能力中的唯一明显缺口。

它的目标是：

> 把模糊入口收成稳定需求边界，防止问题直接滑进方案设计或实施。

## 2. 当前最小结构

结合第一轮外部对标，当前更适合把 `需求收敛` 收成一个一级能力簇，内部固定为三个子阶段：

1. `入口分流`
2. `需求定义`
3. `Ready Gate`

## 3. Step 1: 入口分流

目标：

- 接住任务入口
- 判断本轮到底在处理什么
- 识别它属于 feature、bug、治理、结构升级还是接入类问题
- 判断它应该进入哪条下游主线，而不是默认滑进 `方案设计`

至少要回答：

1. 这次要处理的核心对象是什么
2. 为什么现在值得处理
3. 它影响什么，不影响什么
4. 当前入口是否已经足够清楚，可以进入下一步

产出：

- 一份被收窄的任务意图
- 一条明确的下游去向判断

## 4. Step 2: 需求定义

目标：

- 把已经收窄的任务意图固定成后续设计可消费的约束面

至少要回答：

1. 本轮必须满足什么
2. 本轮明确不做什么
3. 什么信号可以说明本轮达成
4. 后续 `方案设计` 的输入边界是什么

产出：

- 稳定的需求边界
- 待进入 `Ready Gate` 的输入面

## 5. Step 3: Ready Gate

目标：

- 判断当前输入是否已经足够进入 `方案设计`
- 如果还不够，明确缺的是哪些前置信息，而不是直接用 spec 去兜底

至少要回答：

1. 当前范围是否已经足够稳定
2. 成功信号是否已经足够明确
3. 是否仍存在会显著影响后续设计的关键未知
4. 是否已具备进入 `方案设计` 的最低条件

产出：

- `可进入方案设计`
  - 或
- `暂不可进入方案设计，需补齐的前置信息`

## 6. 进入条件

进入 `需求收敛` 前，至少应满足：

1. 已有 issue、想法、问题、反馈或接入任务入口
2. 当前还不能直接进入 `方案设计`

## 7. 退出条件

退出 `需求收敛` 后，至少应满足：

1. 已能清楚说明“这次到底要改什么”
2. 已能清楚说明“这次必须满足什么”
3. 已能清楚说明“这次明确不做什么”
4. 已完成 `Ready Gate`
5. 后续可以稳定进入 `方案设计`

## 8. 当前承接策略

当前不建议立即把 `需求收敛` 固化成一个新 skill 名称。

更合理的短期策略是：

1. 先把它作为固定 workflow 段存在
2. 在主流程上明确它位于：
   - `现状同步 -> 需求收敛 -> 方案设计`
3. 后续如果输入输出长期稳定，再评估是否产品化为独立显性对象

## 9. 当前结论

`需求收敛` 现在已经不再只是一个抽象缺口，而是可以被明确表达为一段前段 workflow。

当前最小结构也已进一步稳定为：

1. `入口分流`
2. `需求定义`
3. `Ready Gate`
