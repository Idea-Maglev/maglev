# skill结构性升级 事实结晶 Workflow 草案

> 状态：草案
> 目标：将 `事实结晶（Crystallization）` 从抽象主流程动作压成可讨论、可编排的 workflow 草案。
> 关联：
> - `02_design.md`
> - `03_plan.md`

## 1. 定位

`事实结晶（Crystallization）` 当前定位为：

- 主流程中的生命周期动作
- 不是立即产品化的独立一级 skill
- 由多个现有对象与 workflow 步骤共同完成的末端闭环

它的核心目标不是“归档”，而是：

> 把已经通过验证、确认成立的变化，沉淀为新的项目现实，并结束本轮进行中状态。

这里需要和 `maglev_archival_check` 明确区分：

- `maglev_archival_check`
  - 更偏思考归档 / 知识沉淀检查
- `事实结晶（Crystallization）`
  - 更偏 Reality 回写 / 生命周期收口

## 2. 前置条件

进入 `事实结晶` 前，至少应满足：

1. 本轮目标已达成，或已明确终止
2. 关键结果已完成验证
3. active 需求已有可结晶的有效产物
4. 已有明确的收口意图，而不是仍处于开放探索中

## 3. Workflow 步骤

### Step 1. 结晶条件确认

目标：

- 确认当前是否具备进入结晶阶段的条件

当前主要承载对象：

- `maglev_archival_check`

补充说明：

- `maglev_archival_check` 当前更偏知识沉淀与归档检查
- 因此这一步仍需要补充人工判断，不能把它直接等同于 Reality 结晶条件判断

需要回答：

- 哪些结果已经成立
- 哪些仍是过程草稿
- 当前 active 内容是否具备收口条件

通过标准：

- 可以明确进入后续结晶步骤
- 不再停留在“差不多完成”的模糊状态

### Step 2. 现实回写判定

目标：

- 判断哪些变化应写回 `10_reality`
- 判断写回位置与写回粒度

当前承载方式：

- `事实结晶` 下的独立 workflow 步骤
- 暂不单独产品化为一级 skill

需要回答：

- 哪些变化已经构成新事实
- 这些新事实应写到哪个 reality 文件或模块分区
- 哪些内容只应保留在过程记录中

通过标准：

- 已明确 Reality 回写边界
- 不再混淆新事实与过程草稿

### Step 3. active 状态收口

目标：

- 判断这条 active 主线是否结束、继续，或拆分

当前承载方式：

- `事实结晶` 下的独立 workflow 步骤
- 暂不单独产品化为一级 skill

需要回答：

- 本轮目标是否完成
- 本轮 active 是否可以结束
- 哪些过程记录需要保留
- 是否要拆出新的 active 主线继续推进

通过标准：

- `20_evolution/active` 中该主题状态明确
- 不再出现“已完成但长期悬挂”的 active 目录

### Step 4. 索引与可发现性回填

目标：

- 让新写回的 Reality 能被后续会话和维护者发现

当前主要承载对象：

- `maglev-map-maker`
- `maglev-librarian`

触发条件：

1. `10_reality` 有新增或重写
2. active 需求产出了新的长期结构对象
3. skill / workflow / 目录结构发生变化
4. 新增了会被高频依赖的知识入口

通过标准：

- 至少存在一个与本轮变化匹配的可发现性回填动作
- 新现实不会因为缺少索引或地图而“写了等于没写”

## 4. 与现有对象的关系

| 对象 | 在结晶 workflow 中的角色 |
| :--- | :--- |
| `maglev_archival_check` | 知识沉淀质量门 / 结晶前检查对象 |
| `10_reality` 更新动作 | 新事实的正式写回位置 |
| `20_evolution/active` 收口动作 | 进行中状态的结束或延续判断 |
| `maglev-map-maker` | 项目结构与全景映射回填 |
| `maglev-librarian` | 索引、导航与知识可发现性回填 |

## 5. 当前结构判断

基于当前讨论，`事实结晶` 更像：

- 一个 workflow
- 一个生命周期末端编排
- 一个跨对象的主流程闭环

而不是：

- 一个新的大而全 skill
- 一个单一对象包办的动作

## 6. 当前结论

当前更推荐的实现方向是：

1. 保持 `事实结晶` 作为正式主流程动作
2. 先把上述 4 步固化为稳定 workflow
3. 再评估其中哪些步骤值得继续产品化

这样可以优先补齐主流程闭环，而不是继续制造新的抽象对象。
