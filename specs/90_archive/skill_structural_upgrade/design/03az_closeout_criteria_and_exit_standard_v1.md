# skill结构性升级 Closeout Criteria 与退出标准 v1

> 状态：已满足，可封板
> 作用：为当前 `skill结构性升级` 主线提供正式退出条件、封板判断与延后事项清单，避免对象级收口后继续无边界漂移。

## 1. 当前定位

这份文档回答的不是“还有没有任何事情能做”，而是：

1. 哪些条件满足后，本轮主线可以正式封板
2. 哪些事项属于下一轮，而不是继续留在当前主线中拖延
3. 当前收口后的仓库应以什么口径被理解

## 2. 当前已满足的封板前提

以下前提当前已经基本满足：

1. `5+3` 顶层能力骨架已稳定
2. 主流程新骨架对象已进入运行面：
   - `entry-router`
   - `knowledge-check`
   - `requirement-convergence`
   - `crystallization`
3. 质量层三面已完成现役收口
4. `skill-scout` / `skill-squadron` / `.agents/private-catalog.yaml` 治理底座已形成
5. `contribute_methodology` 已清理
6. `mermaid-expert` 已完成命名收口
7. `maglev-create-prd` 已吸收到 `requirement-convergence` 并退出运行面
8. spec pipeline 四件套已明确降为内部模块链，不再按独立现役 skill 解释

## 3. 正式退出条件

当以下条件同时成立时，本轮 `skill结构性升级` 可以视为达到可封板状态：

1. 现役运行面不再存在明显冲突的旧入口或重复对象
2. 主流程核心对象、质量层对象与治理底座对象的现役口径稳定
3. `.agents/`、主线设计文档、现状文档与对外能力快照之间不存在明显互相矛盾
4. 非主干对象已完成“清理 / 豁免 / 去显性化 / 延后”的分流判断
5. 对象级最终复核清单不再存在阻塞项
6. 退出标准本身已有独立落点，而不是继续散落在阶段总结里

## 4. 当前仍保留但不阻塞封板的事项

以下事项可以继续保留，但不应阻止本轮封板：

1. 主流程核心对象的 rename 策略是否还要单独出文档
2. `.agents/private-catalog.yaml` 后续是否继续扩容更多专项支持对象
3. 更多专项支持对象是否要进入下一轮治理

这些事项属于：

- 后续维护
- 下一轮治理
- 或新一轮 `skill-scout` / `skill-squadron` 任务

而不属于当前主线继续展开的理由。

## 5. 当前不应再继续扩写的方向

本轮封板前，不再继续做：

1. 回到历史大讨论重新争论骨架
2. 为了“完整性”新增更多分析稿
3. 把所有内部模块立刻物理重构
4. 扩写新的对象候选，而不先完成封板

## 6. 当前建议结论

当前主线已经满足封板条件。

更合理的动作顺序应是：

1. 用本文件作为退出标准落点
2. 在 `03_plan.md`、`04_stage_summary.md`、最终复核清单中引用本文件
3. 确认现役运行面与文档口径无明显冲突后，宣布本轮结构升级封板

当前执行结果：

- 现役运行面已无 `maglev-create-prd` 冲突入口
- spec pipeline 四件套已统一降为内部模块链口径
- 主线入口、设计入口、阶段总结与退出标准已互相挂接

因此本文件当前可作为本轮封板依据直接使用。

## 7. 封板后的下一轮入口

本轮封板后，后续若继续推进，应进入新主题，而不是继续挂在当前主线下。

更合理的新主题包括：

1. rename 策略专项收口
2. 新一轮治理对象准入控制
