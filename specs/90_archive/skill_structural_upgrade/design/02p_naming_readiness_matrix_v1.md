# skill结构性升级 定名准备度矩阵 v1

> 状态：进行中
> 作用：判断当前各类对象是否已经接近正式定名，还是只能先保留结构语义。

## 1. 目标

当前主线已经不再缺结构判断，真正需要防止的是：

- 结构刚收清，就急着起新名字
- 名字先固定，结果后面结构又改
- 把 `Workflow-first` 对象误当成已经可以产品化的正式 skill

所以这里需要一份“定名准备度”矩阵。

## 2. 判断标准

一个对象若要进入“可正式定名”状态，至少应满足：

1. 结构位置已经稳定
2. 生命周期边界已经稳定
3. 输入输出已经足够清楚
4. 不会持续吞并相邻阶段
5. 命名风险模型中不存在明显高风险未处理项

补充说明：

这里的“可正式定名”，当前优先指：

- `formal_action_name`

不等于：

- `canonical_skill_name` 已必须同步定稿

## 3. 准备度分级

当前统一分为三档：

### A. `可正式定名`

含义：

- 可以开始把结构动作名提升为正式稳定名称
- 后续主要是表达优化，不太会再改对象边界
- 但未来正式 skill 名可继续单独讨论

### B. `可先定结构语义，不宜急着定正式名`

含义：

- 结构判断已经成立
- 但对象形态、边界或最终承接方式仍可能变化
- 当前更适合保留“结构动作名 / 能力名”，不急着产品化

### C. `旧对象应移除`

含义：

- 当前对象已被确认具有高误导性
- 不再保留为运行面入口
- 只在历史分析文档中保留提及

## 5. 当前最关键判断

当前已经接近正式定名的，主要是主流程中段 4 个对象：

1. `现状同步`
2. `方案设计`
3. `上下文实施`
4. `综合验证`

它们现在更适合：

- 先稳定结构动作名
- 再决定未来是否需要真的改文件夹名

而当前最不应急于定名的，是：

1. `需求收敛`
2. `现实结晶`
3. `整体接入`
4. `能力进化`
5. `现状表达`

原因很直接：

- 它们的结构语义已经成立
- 但对象形态仍不是单点 skill

## 6. 当前结论

后续如果进入命名收口阶段，当前建议遵循这条顺序：

1. 先收中段主流程对象的正式动作名
2. 再单独判断未来正式 skill 名，而不是要求它和动作名一一拟合
3. 再维持前段缺口和后段闭环的结构语义，不急于产品化命名
4. 已确认误导性强的旧对象直接移除，不再保留运行面兼容写法
