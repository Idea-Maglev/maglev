# skill结构性升级 判据 v1

> 状态：进行中
> 作用：将当前已形成的主线判断、巡逻结果与来源校准压成一套可复用的统一判据，作为 Maglev 后续反省、治理与进化的依据。

## 1. 使用目标

这份判据不负责替代详细分析，而负责在后续判断任意 skill / workflow / 结构对象时，先回答：

1. 它是否真的承接 Maglev 要解决的问题
2. 它应以什么形态存在
3. 它当前最合理的处理方向是什么

## 2. 总前提

后续所有结构判断，都应先满足以下前提：

1. 目标不是修现有 skill 的局部病灶
2. 目标是根据 Maglev 要解决的问题，构建最合适的 skill 体系
3. 现有对象的问题主要作为反证材料，而不是主线本身

## 3. 证据顺序

后续外部对标与结构判断，应按以下证据顺序执行：

1. **第一层证据**
   - 私有资源池命中的来源
   - 来源于：
     - `.agents/skills/skill-scout/references/source-registry.yaml`
     - `skill-sources.yaml`
     - `.agents/skills/skill-scout/references/user-source-preferences.yaml`
2. **第二层证据**
   - 官方公开资料
   - 用于校正、补强或复核
3. **第三层证据**
   - 仓库历史对象与当前存量实践
   - 仅作为现实约束与迁移成本参考

补充执行门槛：

- 当工作还处于结构判断、命名探索、元数据设计阶段时，可在当前 active spec 内推进
- 当工作进入真正的 skill 撰写、改写或基于外部高相似度 skill 的适配落地时，必须切换到 `skill-scout` 机制执行
- 当工作进入 `skill-scout` 机制后，若没有完成实际联网检索并留下显式证据，则不得进入 `evaluate`、`adapt`、`register`

补充强制要求：

- 仅有来源池映射，不算完成 `search`
- `search` 产物中必须显式记录联网校验链接
- 任何没有联网证据的 skill 生成结果，默认视为不可信，不应进入运行面

## 4. 顶层能力判据

一个对象若要成为未来 Maglev 的用户显性一级能力，至少应满足：

1. 它承接的是稳定能力断点，而不是临时补丁
2. 它不是别的一级能力的内部实现链一段
3. 它有清晰的输入、输出和进入条件
4. 它不会持续吞并相邻生命周期阶段

当前用户显性一级能力骨架为：

1. `现状同步`
2. `需求收敛`
3. `方案设计`
4. `上下文实施`
5. `综合验证`

当前体系级能力骨架为：

1. `整体接入`
2. `能力进化`
3. `现状表达`

## 5. 生命周期判据

后续所有对象都必须先回答：

1. 它服务哪条生命周期链
2. 它位于该链的哪个阶段
3. 它是否错误占用了别的链的语义入口

当前最关键的生命周期分离是：

1. `思考归档触发链`
   - 面向知识沉淀
   - 可重复触发
2. `事实结晶触发链`
   - 面向 Reality 回写与 active 收口
   - 面向后段状态转换

这两条链不能共用“归档”作为默认语义入口。

## 6. 形态判据

判断一个能力应该以什么形态存在，当前优先顺序为：

1. 先判它是不是顶层能力
2. 再判它更适合：
   - 显性 skill
   - workflow 段
   - 内部模块
   - 质量能力面
   - 协作承接簇
   - 底层长期能力

当前已较稳定的形态结论：

### A. 适合 `Workflow-first`

1. `需求收敛`
   - 最小结构：
     - `入口分流`
     - `需求定义`
     - `Ready Gate`
2. `现实结晶`
   - 最小结构：
     - 结晶条件确认
     - 现实回写判定
     - active 状态收口
     - 索引与可发现性回填

### B. 适合 `Merge`

质量层对象簇应收成三面：

1. `Spec Audit Surface`
2. `Review / Validation Surface`
3. `Test Design Surface`

### C. 适合 `协作承接`

体系级能力当前继续采用协作承接：

1. `整体接入`
2. `现状表达`
3. `能力进化`

## 7. 命名判据

后续命名必须优先满足：

1. 低歧义
2. 中性
3. 行业可对齐
4. 能直接提示结构职责

当前高风险命名模式包括：

1. 跨生命周期语义吞并
2. 错误价值重心前置
3. 动作语义与结构角色混写
4. 结构重要性被轻量词压低

## 8. 当前稳定方向

基于当前四轮巡逻与来源校准，以下方向已足够稳定，可作为后续治理默认前提：

1. `需求收敛` 继续保持 `Workflow-first`
2. `现实结晶` 与 `思考沉淀` 继续严格拆分
3. `maglev_archival_check` 更接近知识沉淀检查对象
4. 质量层继续保持三面结构，不回退为并列 skill 平铺
5. `maglev-cross-validate` 继续作为主流程汇聚点，而不是质量层总 skill
6. `整体接入` / `现状表达` / `能力进化` 继续保持能力层显性、对象层协作承接

## 9. 当前不应做的事

在没有新证据推翻当前判据前，后续不应：

1. 继续把 `create-spec` 当作前段总入口
2. 把 `现实结晶` 和 `思考沉淀` 重新合成一个“归档”大词
3. 把质量层重新铺成一串并列 skill
4. 为了“简洁”把体系级能力簇强收成单点总 skill
5. 绕过私有资源池，直接把公开资料当成第一入口证据

## 10. 当前结论

这份判据 v1 的意义，不在于它已经给出所有最终裁决，而在于：

- 它已经足以约束后续大多数升级讨论
- 它可以作为 `skill-scout` / `skill-squadron` 的统一判断底座
- 它是当前 Maglev 反省与进化最值得保留的一层高价值资产
