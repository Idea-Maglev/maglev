# skill结构性升级 正式动作名定稿草案 v1

> 状态：进行中
> 作用：为当前已接近正式定名的主流程对象给出第一版稳定动作名，用于后续文档、README、Reality、元数据与治理口径统一。

## 1. 使用范围

这份草案当前只处理：

- 已进入“可正式定名”状态的主流程对象

当前不处理：

- `需求收敛`
- `现实结晶`
- `思考沉淀`
- `整体接入`
- `能力进化`
- `现状表达`

原因：

- 这些对象当前仍更适合保留结构语义
- 不宜过早产品化成正式 skill 名称

## 2. 定稿原则

本轮“正式动作名定稿”仅意味着：

1. 后续结构文档优先采用该动作名
2. README / Reality / 元数据优先采用该动作名
3. 历史 skill 名继续保留为兼容写法
4. 未来若重建或重命名 skill，skill 名称不必与正式动作名一一相同

本轮不自动意味着：

1. 立即改 skill 文件夹名
2. 立即改全部历史入口
3. 立即停止使用旧名
4. 未来 skill 名必须继续拟合历史对象名

补充原则：

- `正式动作名`
  - 服务结构表达
- `未来 skill 名`
  - 服务对象分发与识别
- `历史兼容名`
  - 服务迁移与兼容

这三者当前应明确分离，而不是继续绑定成同一个名字体系。

## 3. 当前定稿对象

### A. `maglev-standup`

历史对象：

- `maglev-standup`

正式动作名：

- `现状同步`

兼容解释：

- `maglev-standup（历史命名，结构上按现状同步理解）`

说明：

- 当前不再优先使用“standup”作为结构动作主名
- `会话启动同步` 可作为辅助解释，不作为顶层主名

### B. `maglev-create-spec`

历史对象：

- `maglev-create-spec`

正式动作名：

- `方案设计`

兼容解释：

- `maglev-create-spec（历史命名，结构上按方案设计理解）`

说明：

- 当前不再让 `create-spec` 继续吞并前段能力
- 它的正式结构职责聚焦于可执行、可验证方案依据的形成

### C. `maglev-quick-dev`

历史对象：

- `maglev-quick-dev`

正式动作名：

- `上下文实施`

兼容解释：

- `maglev-quick-dev（历史命名，结构上按上下文实施理解）`

说明：

- 当前不再把“quick”当作正式价值重心
- 该对象的结构重点是基于清晰前置与上下文完成实施

### D. `maglev-cross-validate`

历史对象：

- `maglev-cross-validate`

正式动作名：

- `综合验证`

兼容解释：

- `maglev-cross-validate（历史命名，结构上按综合验证理解）`

说明：

- 当前继续保留其主流程汇聚点定位
- 不把它改写成质量层总 skill

## 4. 当前推荐写法

后续文档中，当前优先推荐以下写法：

1. `现状同步（maglev-standup）`
2. `方案设计（maglev-create-spec）`
3. `上下文实施（maglev-quick-dev）`
4. `综合验证（maglev-cross-validate）`

如果上下文已经足够明确，也可以直接只写：

1. `现状同步`
2. `方案设计`
3. `上下文实施`
4. `综合验证`

## 5. 当前结论

这一版定稿草案的意义，不是宣告旧名立刻失效，而是先稳定一层更适合未来 Maglev 的正式结构语言。

后续如果继续进入重命名阶段，应优先遵循：

1. 先统一结构文档与元数据口径
2. 再决定是否需要物理重命名
3. 继续保留历史对象的兼容写法，直到迁移成本可接受
