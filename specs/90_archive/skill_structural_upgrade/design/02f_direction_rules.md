# skill结构性升级 处理方向规则

> 状态：草案
> 作用：为批量分析底表中的 `current_direction` 提供统一处理规则。

## 1. 目标

当前批量分析底表已经能给出对象的初步处理方向，但如果没有统一规则，后续仍然会出现：

- 同样写着 `Keep`，实际含义不一样
- 同样写着 `Rename`，处理方式完全不同
- `Workflow-first` 和 `Demote` 边界漂移

所以这里需要把处理方向收成可执行规则。

## 2. 处理方向清单

当前统一使用 6 种处理方向：

1. `Keep`
2. `Rename`
3. `Demote`
4. `Merge`
5. `Workflow-first`
6. `Observe`

## 3. 规则定义

### 3.1 `Keep`

含义：

- 当前对象继续保留为正式对象

适用条件：

1. 已较稳定承接某个能力断点
2. 没有明显结构错位到必须立即改名或降级
3. 即使有表达问题，也更适合先补解释而不是动对象形态

标准动作：

1. 保留对象
2. 必要时补正式动作名或结构解释
3. 不立即动文件夹或分发形态

### 3.2 `Rename`

含义：

- 当前对象能力保留，但名称不再适合作为正式结构名

适用条件：

1. 名称持续误导智能体或用户
2. 名称占据了错误语义入口
3. 名称与对象真实职责长期不匹配

标准动作：

1. 保留能力
2. 先补新的正式动作名或正式对象名
3. 允许短期保留历史命名作为兼容层
4. 后续再决定是否改文件夹名与入口名

### 3.3 `Demote`

含义：

- 当前对象仍有价值，但不应继续作为显性并列对象存在

适用条件：

1. 它本质是内部模块
2. 它更像某个一级对象的实现链一段
3. 继续并列暴露只会放大认知负担

标准动作：

1. 保留其功能
2. 在结构认知上降为内部模块或二级能力
3. 后续说明文档中不再把它当一级心智单元强调

### 3.4 `Merge`

含义：

- 当前对象和相邻对象应被收进同一能力面或同一对象簇

适用条件：

1. 多个对象职责高度重叠
2. 它们继续平铺只会制造碎片化
3. 更合理的结构是“一个能力面 + 若干内部专项”

标准动作：

1. 先识别要并入的目标能力面
2. 保留专项差异，但不再维持并列一级心智
3. 后续可逐步统一入口和说明

### 3.5 `Workflow-first`

含义：

- 当前对象或能力先不产品化为独立 skill，而优先以 workflow 段存在

适用条件：

1. 能力真实存在，但输入输出还不够稳定
2. 更像一个生命周期动作或阶段性编排
3. 过早对象化会制造新的抽象负担

标准动作：

1. 先在主流程或专项流程中固定其位置
2. 明确进入条件、退出条件、上下游边界
3. 后续如果长期稳定，再评估是否独立对象化

### 3.6 `Observe`

含义：

- 当前先不裁决，保持观察

适用条件：

1. 结构价值存在，但当前问题还不足以做方向性裁决
2. 还缺上下文或还在等待更上位结构定稿
3. 过早裁决可能导致误伤

标准动作：

1. 记录当前风险
2. 暂不进入重写或重命名
3. 在后续结构更稳定后再复判

## 4. 当前边界判断

最容易混淆的两组方向是：

### A. `Demote` vs `Workflow-first`

- `Demote`
  - 对象仍存在，但层级下降
- `Workflow-first`
  - 能力先不对象化，优先以流程段存在

### B. `Keep` vs `Observe`

- `Keep`
  - 当前方向已经足够清楚
- `Observe`
  - 当前仍不建议给出明确动作方向

## 5. 当前结论

这套规则表的作用，不是替代对象分析，而是让批量分析底表中的处理方向拥有统一含义。

后续 `skill-squadron` 若做批量巡逻，应优先复用本表来解释每个对象的 `current_direction`。
