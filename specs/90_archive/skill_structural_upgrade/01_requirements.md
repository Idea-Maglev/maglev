# skill结构性升级 Requirements

> 状态：进行中
> 类型：结构治理 / 能力体系演进
> 关联：
> - `00_intent.md`
> - `00_context.md`

## 1. 需求目标

本轮需求要解决的不是单个 skill 的内容质量，而是 Maglev 当前 skill 体系与主流程之间的结构错位问题。

进一步说，本轮不应以“修复当前 skill 的病灶”为目标，而应以“根据 Maglev 要解决的问题，构建最合适的 skill 体系”为目标。

本轮产出应支持以下结果：

1. 能用统一结构解释 Maglev 当前主流程
2. 能说明 `issue`、`intent`、`evolution`、`reality` 的关系
3. 能在此基础上重新判断 skill 的分层与职责
4. 能为后续 skill 升级提供稳定的结构依据，而不是继续增量打补丁
5. 能把现有 skill 的问题降级为“反证材料”，而不是把逐个修补当作主线

## 2. 需要满足的核心要求

### 2.1 主流程要求

需要重新识别并明确 Maglev 当前主流程，至少回答：

- 一个需求从触发到进入执行，经过哪些必要阶段
- 哪些阶段是正式阶段，哪些只是辅助动作
- 哪些阶段应该有独立承载层

### 2.2 生命周期边界要求

需要明确以下对象在主流程中的职责边界：

- `issue`
- `intent`
- `specs/20_evolution/active/`
- `10_reality`

至少应回答：

- 什么内容属于任务入口
- 什么内容属于进行中的演进正文
- 什么内容属于已成立事实
- 哪些动作属于“思考归档触发链”，哪些属于“事实结晶触发链”
- 不同生命周期动作是否错误共用了同一个高层语义词

### 2.3 `intent` 产物要求

需要判断 `intent` 是否应成为正式阶段，并明确：

- 是否需要独立物理文件
- 如果独立，应承担什么后续作用
- 它与 `requirements`、`spec`、`validate`、`crystallize` 的关系是什么

### 2.4 skill 结构要求

需要在主流程和生命周期边界清晰之后，重新定义 skill 体系至少包括：

- 哪些能力属于主流程核心层
- 哪些能力属于系统基础设施层
- 哪些能力属于进化治理层
- 哪些能力属于专项支持层

### 2.5 workflow 关系要求

需要明确 workflow 与 skill 的标准关系，至少回答：

- workflow 是否只是用户入口包装层
- skill 是否才是真正能力承载层
- 哪些 workflow 应与 skill 一一映射，哪些不必强制对等

### 2.6 元数据要求

需要形成一组可执行的最小元数据要求，用于后续治理、分发与升级判断。

这组元数据至少应能支持：

- 生命周期状态判断
- 能力层级判断
- 入口关系判断
- 分发边界判断

### 2.7 资产化要求

本轮分析结果需要被视为 Maglev 的长期进化资产，而不是一次性讨论副产物。

至少应满足：

- 能作为后续 skill 反省与升级的依据
- 能被 `skill-scout` / `skill-squadron` 复用
- 能保留判断形成的上下文、边界与修正线索
- 能支持后续对当前结论进行复核、增补或推翻

### 2.8 `skill` 撰写门槛要求

当后续工作进入以下动作时，不应继续直接在当前讨论链中拍板生成 skill 正文，而应切换到 `skill-scout` 机制：

- 新建一个新 skill
- 重写某个既有 skill 的核心定义
- 为某个对象正式定稿 `canonical_skill_name` 并准备落地
- 基于外部高相似度 skill 做适配式撰写

也就是说：

- 结构讨论
- 判据沉淀
- 元数据设计
- 命名探索

可以先在当前 active spec 中推进；

但一旦进入真正的 skill 撰写与落地阶段，就必须通过 `skill-scout` 的来源加载、候选对比、评估与适配流程执行。

进一步说，这里的 `skill-scout` 不是形式上的“参考一下”。

至少必须满足：

- 已实际联网搜索高相似度可参考对象
- 已在 `search` 产物中记录联网校验链接
- 已基于这些链接完成 `evaluate`

若上述条件不满足，则本轮 skill 生成结果不应被视为合格产物。

## 3. 约束

### 3.1 范围约束

本轮不要求：

- 立即重写全部 skill
- 逐个优化现有 skill 内容
- 直接做全面 skill 裁撤
- 在流程未对齐前先决定所有 skill 的最终去留

### 3.2 结构约束

本轮讨论与产出应优先沉淀在：

- `specs/20_evolution/active/skill_structural_upgrade/`

而不是将进行中的需求正文继续主要挂在 `issues/` 中。

### 3.3 表达约束

本轮 requirements 应聚焦：

- 要满足什么
- 要回答什么
- 要形成什么边界

而不继续承载大段讨论过程、推理过程和来回修正记录。

这些内容应优先进入：

- `00_context.md`

## 4. 当前阶段预期产物

本轮至少需要形成以下产物：

1. 一份可持续追加的 `00_context.md`
2. 一份可约束后续阶段的 `00_intent.md`
3. 一份聚焦核心要求与边界的 `01_requirements.md`
4. 后续可继续演进的结构设计文档

## 5. 验收信号

当以下条件成立时，可认为本轮 requirements 阶段达成基本目标：

- 能清楚说明进行中需求应落在 `specs/20_evolution/active/` 而不是主要停留在 `issues/`
- 能清楚说明 `intent` 的职责、后续作用和物理承载方式
- 能以主流程为基准重新讨论 skill 分层，而不是继续按现有目录并列理解
- 当前 active spec 内部文件分工已开始体现这种新边界
