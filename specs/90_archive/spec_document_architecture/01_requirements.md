# Requirements — Spec 文档体系设计

## 来源

- 2.14 复盘 P0-3：**设计稿→代码无 Skill 支撑**（董帅/赵轩）
- Kiro 模板分析中发现的前端/UI 覆盖缺口
- `main_flow_quality_gates` spec 讨论中拆分出的文档架构决策

## 范围说明

> 本 spec 聚焦**文档体系扩展**：交互需求/设计文档拆分、术语表机制、文档互联互验、UX 技能定位。
>
> **前置依赖**：`main_flow_quality_gates` 的 F-2（结构化需求格式）和 F-3（AC 追溯链）提供编号基础。

---

## 文档结构决策

> 需求文档和设计文档均按**功能/行为**与**交互/视觉**拆分为两份独立文件。
> 拆分原因：模板和示例统一偏后端会导致前端/UI 需求被静默跳过（2.14 P0-3 的根因之一）。

```
spec/{feature}/
├── 01_requirements_functional.md       # F-{N}, AC-F{N}-{M}
├── 01_requirements_interaction.md      # I-{N}, AC-I{N}-{M}
├── 02_design.md                        # 技术设计
├── 02_design_interaction.md            # 交互设计
├── 03_plan.md                          # 统一实施计划
└── context/                            # 审批记录、参考资料、UX 研究产物
```

## maglev-design-ux 定位决策

> **决策**：保留 `maglev-design-ux` 技能，重新定位为 spec-designer 的**可选上游输入**。
>
> **原定位**：独立产出 `02_ui_design.md`，与 spec-designer 平行。
> **新定位**：产出 Persona + User Journey + 情绪曲线，存入 `context/`，作为 spec-designer 生成 `02_design_interaction.md` 时的输入。
>
> **理由**：
> - UX 技能的 Persona/Journey 是用户研究能力，新交互模板是工程交付格式——两者不在同一阶段
> - 新交互模板已覆盖 UI 状态图、组件 API、响应式、可访问性、Design Token——UX 技能中这些维度要么缺失要么仅隐含
> - 保持 UX 技能独立可避免 spec-designer 职责膨胀
>
> **流程**：
> ```
> [requirement-convergence] → 01_requirements_*.md
>                              ↓
>                         [maglev-design-ux]（可选）→ context/persona.md + context/journey.md
>                              ↓ 作为输入
>                         [spec-designer] → 02_design.md + 02_design_interaction.md
> ```

## 功能需求 F-1: 交互需求文档结构

**用户故事**: 作为前端开发者或 UI 设计协作者，我希望交互/视觉需求有独立的结构化文档，以便在设计和实现 UI 时有明确的行为和视觉基线

**验收标准**:

- AC-F1-1: `01_requirements_interaction.md` 应包含以下必须小节：概述、交互需求列表（I-{N}）、术语表
- AC-F1-2: 每条交互需求（I-{N}）应包含：交互场景描述 + 至少 1 条 AC（AC-I{N}-{M}）
- AC-F1-3: 交互需求 AC 应覆盖以下至少一个维度：UI 状态（加载/空/错误/成功/骨架屏）、用户操作响应（点击/输入/手势/键盘）、视觉约束（响应式断点/布局切换/动效要求）、可访问性要求
- AC-F1-4: 当交互需求依赖某个功能需求的完成结果时，应在 AC 中显式引用对应的 F 系 AC（如"当 AC-F2-1 的表单提交成功后，界面应..."）

**边界情况**:
- 纯后端项目（API 服务、CLI 工具、数据管道）不生成此文档
- 如果交互需求非常简单（如仅一个确认弹窗），可作为功能需求的附属 AC 放在 `01_requirements_functional.md` 中，不必拆分

**模板预览**: `context/requirement_interaction_template_preview.md`

## 功能需求 F-2: 术语表机制

**用户故事**: 作为进入项目的新成员或 AI 助手，我希望项目中有统一的术语定义，以便快速理解项目语境中的专有名词和缩写

**验收标准**:

- AC-F2-1: 当 requirement-convergence 生成结构化需求文档时，文档应包含「术语表」小节，定义该功能涉及的关键术语
- AC-F2-2: 在 reality 层应维护一份项目级术语文件（如 `glossary.md`），汇总跨功能的项目术语和黑话
- AC-F2-3: 当需求文档中出现术语表已收录的术语时，应与项目级术语定义保持一致；若有歧义，应在需求文档中显式说明差异
- AC-F2-4: 术语表中每条术语应包含：术语名 + 定义，消除同一个词在不同上下文中被不同理解的风险

**边界情况**:
- 简单任务不生成结构化需求文档时，不要求术语表
- 项目级术语文件是增量维护的，不要求一次性补全所有术语

## 功能需求 F-3: 文档互联互验机制

**用户故事**: 作为使用拆分文档的开发者，我希望功能文档和交互文档之间有清晰的关联和同步机制，以便在修改一份文档后不会遗漏对另一份的连锁更新

**验收标准**:

- AC-F3-1: 每份需求和设计文档的头部应包含「文档关系」小节，声明其依赖的文档和被消费的下游文档
- AC-F3-2: 当 integrated-validator 执行审计时，应同时扫描 F 系和 I 系需求文档，检查两系 AC 是否均被对应设计文档覆盖
- AC-F3-3: 03_plan.md 的任务列表应同时引用 F 系和 I 系 AC，不允许某一系的 AC 在计划中完全缺失
- AC-F3-4: 当 F 系需求文档发生变更时，系统应提示用户检查 I 系需求文档和两份设计文档是否需要同步更新

**边界情况**:
- 仅有功能需求文档（无交互文档）的项目，互联验证退化为单文档模式
- 「提示用户检查」为最小实现——不要求自动修改关联文档，但必须可见地提醒

**模板预览**: `context/design_interaction_template_preview.md`

## 非功能需求

### NFR-1: 向后兼容
纯后端项目不强制生成交互文档。不生成结构化需求文档时，所有下游行为与当前一致。

### NFR-2: 纯 Markdown 输出
所有文档输出均为纯 Markdown 格式。文档数量按项目类型弹性调整（纯后端 2 份、含 UI 最多 4 份需求+设计文档）。

### NFR-3: 拆分文档的内聚性
每份拆分文档应独立可读——不依赖另一份文档的内容才能理解自身。交叉引用使用 AC 编号（非内容复制），避免信息冗余导致的一致性风险。

### NFR-4: 术语可发现性
项目级术语文件应放在 AI 和人类都容易发现的位置（reality 层），新会话启动时应能被自动加载或提示加载。
