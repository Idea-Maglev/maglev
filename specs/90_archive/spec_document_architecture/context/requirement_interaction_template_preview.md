# 交互需求产物输出模板（01_requirements_interaction.md）

> 本文件展示拆分后交互/视觉需求文档的完整格式。
> 用途：确认模板稳定性后，作为需求产物格式改动的依据。
>
> **来源说明**：
> - 文档拆分决策：用户架构决策（方案 B），解决模板全偏后端导致前端/UI 需求被静默跳过的问题
> - 基础结构沿用功能需求模板：EARS 关键词、AC 编号、术语表
> - I 系编号（AC-I{N}-{M}）：Maglev 自行设计，与 F 系并行
> - 交互维度（UI 状态、操作响应、视觉约束、可访问性）：综合行业通用 UI 需求维度
> - Kiro 原始设计中无交互需求相关内容

---

## 模板结构说明

交互需求文档与功能需求文档平行，覆盖以下维度：

| 维度 | 关注点 | 典型 AC 模式 |
|------|--------|-------------|
| **UI 状态** | 加载/空/错误/成功/骨架屏 | 当 {数据状态} 时，界面应显示 {状态} |
| **操作响应** | 点击/输入/手势/键盘 | 当用户 {操作} 时，界面应 {反馈} |
| **视觉约束** | 响应式/布局/动效 | 在 {屏幕尺寸} 下，{组件} 应 {布局行为} |
| **可访问性** | 键盘/屏幕阅读器/对比度 | 当使用 {辅助技术} 时，{元素} 应 {行为} |

**何时不需要此文档**：纯后端服务、CLI 工具、数据管道等无 UI 项目。

---

## 交互需求文档模板

```markdown
# 交互需求文档

## 文档关系

- **功能需求**: `01_requirements_functional.md`（本文档中的交互需求可引用 F 系 AC 作为前置条件）
- **交互设计**: → `02_design_interaction.md`（消费本文档的 I 系 AC）
- **技术设计**: → `02_design.md`（可能引用本文档中的可访问性和性能约束）

## 简介

{从用户交互视角描述本功能的核心体验目标：用户看到什么、怎么操作、什么感受}

## 术语表

- **{术语 A}**: {定义}
- **{术语 B}**: {定义}

## 交互需求

### 交互需求 1: {场景名称}（I-1）

**交互场景**: {用户在什么场景下进行什么交互操作}

#### 验收标准

1. `AC-I1-1` 当 {UI 状态/用户操作} 时，界面应 {视觉/行为响应}
2. `AC-I1-2` 当 {触发条件} 时，{组件} 应 {交互行为}
3. `AC-I1-3` 在 {环境条件} 下，{元素} 应 {适配行为}

#### 边界情况

- {极端交互场景} → {预期处理}

### 交互需求 2: {场景名称}（I-2）

…（同上结构）
```

### 格式要点

| 元素 | 规则 |
|------|------|
| 需求编号 | `I-{N}`，从 1 开始递增（与 F 系独立编号） |
| AC 编号 | `AC-I{N}-{M}`，全局唯一，与 F 系不冲突 |
| EARS 关键词 | 同功能需求：当…时 / 若…则 / 在…期间 / 应 |
| 交互场景 | 替代"用户故事"，直接描述用户与界面的交互上下文 |
| F 系引用 | 交互 AC 可引用功能 AC 作为前置条件（如"当 AC-F1-1 的登录成功后"） |
| 文档关系 | 必须包含。声明依赖和下游，用于互联验证 |
| 术语表 | 必须包含。侧重 UI/UX 领域术语 |

## 实际输出示例：用户认证模块（交互层）

> 与 `requirement_template_preview.md` 的用户认证示例互补。
> 功能需求定义"系统做什么"，交互需求定义"用户看到什么、怎么操作"。

# 交互需求文档

## 文档关系

- **功能需求**: `01_requirements_functional.md`（邮箱密码登录 F-1、验证码登录 F-2、密码重置 F-3）
- **交互设计**: → `02_design_interaction.md`
- **技术设计**: → `02_design.md`

## 简介

用户认证模块的交互体验设计。核心目标：让用户在登录/注册/重置流程中始终清楚"当前在哪一步、发生了什么、下一步该做什么"。三种登录方式（邮箱密码/手机验证码/密码重置）共享统一的视觉语言和交互模式。

## 术语表

- **骨架屏 (Skeleton)**：页面加载中用灰色占位块模拟真实布局的过渡状态
- **内联校验 (Inline Validation)**：用户输入时实时在输入框旁显示校验结果
- **Toast**：页面顶部/底部短暂出现的消息提示，自动消失
- **防抖 (Debounce)**：用户停止输入后延迟一段时间再触发校验，避免频繁请求

## 交互需求

### 交互需求 1：登录表单交互（I-1）

**交互场景**：用户进入登录页面，在邮箱/密码输入框中输入凭证并提交登录

#### 验收标准

1. `AC-I1-1` 当用户进入登录页但尚未操作时，界面应显示空表单状态：邮箱和密码输入框带 placeholder、登录按钮为禁用态（灰色）
2. `AC-I1-2` 当用户在邮箱输入框中输入文本并失焦时，界面应进行邮箱格式内联校验；若格式错误，输入框下方应显示红色提示"请输入有效的邮箱地址"
3. `AC-I1-3` 当邮箱和密码均非空且邮箱格式正确时，登录按钮应自动切换为可用态（主色调）
4. `AC-I1-4` 当用户点击登录按钮后等待响应期间，按钮应切换为 loading 态（显示旋转图标 + "登录中…"文字），且表单不可重复提交
5. `AC-I1-5` 当 AC-F1-1 的登录成功后，界面应跳转到目标页面，跳转前显示成功 Toast
6. `AC-I1-6` 当 AC-F1-2 的登录失败时（凭证错误），界面应在表单上方显示错误横幅"邮箱或密码错误"，密码输入框自动清空并获焦
7. `AC-I1-7` 当 AC-F1-3 的账户锁定时，界面应显示锁定提示"账户已锁定，请 N 分钟后重试"，并附带倒计时显示

#### 边界情况

- 用户在 loading 态刷新页面 → 恢复为空表单初始状态
- 网络断开后点击登录 → 显示"网络连接失败，请检查网络后重试"

### 交互需求 2：验证码登录交互（I-2）

**交互场景**：用户切换到手机验证码登录方式，输入手机号、获取验证码、输入验证码完成登录

#### 验收标准

1. `AC-I2-1` 当用户切换到验证码登录 tab 时，界面应显示手机号输入框和"获取验证码"按钮，验证码输入框隐藏
2. `AC-I2-2` 当用户点击"获取验证码"且 AC-F2-1 的短信发送成功后，按钮应切换为 60 秒倒计时状态（"N 秒后重新获取"），同时验证码输入框出现并自动获焦
3. `AC-I2-3` 当倒计时结束时，按钮应恢复为"重新获取验证码"可点击状态
4. `AC-I2-4` 当 AC-F2-3 的频率限制触发时，界面应显示 Toast "请求过于频繁，请稍后再试"
5. `AC-I2-5` 当用户输入完 6 位验证码时，系统应自动提交（不需要点击登录按钮）

#### 边界情况

- 倒计时期间切换到其他登录方式再切回 → 倒计时继续（不重置）
- 用户修改手机号后重新获取 → 倒计时重置

### 交互需求 3：密码重置交互（I-3）

**交互场景**：用户通过"忘记密码"入口进入多步骤密码重置流程

#### 验收标准

1. `AC-I3-1` 当用户点击"忘记密码"时，界面应导航到密码重置页，显示第一步：输入邮箱
2. `AC-I3-2` 当 AC-F3-1 的重置邮件发送成功后，界面应显示第二步：提示"重置链接已发送到 x**@example.com"（邮箱部分脱敏），并提供"重新发送"链接
3. `AC-I3-3` 当用户通过邮件链接进入新密码设置页时，界面应显示第三步：新密码输入框 + 确认密码输入框 + 密码强度指示器
4. `AC-I3-4` 当 AC-F3-3 的密码重复检测触发时，界面应在新密码输入框下方显示"不能使用最近使用过的密码"

#### 边界情况

- 重置链接已过期时直接访问 → 显示"链接已失效，请重新申请"+ "重新申请"按钮

### 交互需求 4：响应式与可访问性（I-4）

**交互场景**：用户在不同设备和使用不同辅助技术时操作认证流程

#### 验收标准

1. `AC-I4-1` 在视口宽度 ≤ 768px 时，登录表单应从居中卡片布局切换为全宽布局，输入框和按钮宽度为 100%
2. `AC-I4-2` 当使用键盘 Tab 导航时，焦点顺序应为：邮箱 → 密码 → 记住我 → 登录按钮 → 忘记密码 → 其他登录方式
3. `AC-I4-3` 当使用屏幕阅读器时，所有错误提示应通过 `aria-live="assertive"` 自动朗读
4. `AC-I4-4` 所有交互元素的颜色对比度应满足 WCAG 2.1 AA 标准（≥ 4.5:1）

## 简单交互说明

当交互需求非常简单时（如仅一个确认弹窗），不必创建独立的交互需求文档。可以直接在功能需求文档中附加交互相关的 AC：

```markdown
### 需求 N: 删除确认（F-N）

**用户故事**: 作为用户，我希望删除操作有确认步骤，以便防止误操作。

#### 验收标准

1. `AC-FN-1` 当用户点击删除按钮时，系统应弹出确认对话框
2. `AC-FN-2` 当用户确认删除时，系统应执行删除并关闭对话框
3. `AC-FN-3` 当确认对话框显示时，界面应聚焦到"取消"按钮（安全默认）
```

此时不需要 `01_requirements_interaction.md`。

## 模板设计原则

1. **与功能需求互补不重复**：交互需求定义"用户看到什么、怎么操作"，功能需求定义"系统做什么"。同一行为只在一处定义
2. **F 系 AC 作为前置引用**：交互 AC 可引用功能 AC 的结果（"当 AC-F1-1 成功后"），避免重新描述系统行为
3. **维度覆盖检查**：每个交互需求应至少覆盖一个维度（UI 状态/操作响应/视觉约束/可访问性）
4. **文档关系必须声明**：头部"文档关系"小节是互联验证的入口
5. **不存在即不生成**：纯后端项目不创建此文档，不强制填"不适用"
