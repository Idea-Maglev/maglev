# 需求产物输出模板（增强版）

> 本文件展示增强后的需求产物在实际输出中的完整格式。
> 用途：确认模板稳定性后，作为需求产物格式改动的依据。
>
> **格式决策**：输出采用纯 Markdown，人类可读优先。EARS 关键词使用中文（当/若/在/应）。
>
> **来源说明**：
> - 基础结构（Introduction → Requirements → User Story → EARS Acceptance Criteria）：Kiro spec-requirements agent 定义（近似源：`notdp/kiro-for-cc`）
> - 术语表：Kiro 模板未显式规定，但 AI 在生成中一致性添加（用户实际使用多个项目均出现）。Maglev 显式规定为必须
> - EARS 中文关键词：用户决策（Kiro 原设计要求保留英文 WHEN/SHALL/IF/THEN）
> - AC-F{N}-{M} 编号：Maglev 自行设计（Kiro 使用简单数字编号）

---

## 模板结构说明

需求产物由两部分组成：

1. **边界收敛摘要**（现有 prd_output_package，YAML 格式保留不变）
2. **结构化需求文档**（新增，纯 Markdown，附在边界收敛之后或独立为需求文档）

当任务简单、不需要行为级 AC 时，只输出边界收敛摘要，不生成结构化需求文档。

---

## 结构化需求文档模板

```markdown
# 功能需求文档

## 简介

{项目/功能的一段话概述：是什么、为什么做、核心约束}

## 术语表

- **{术语 A}**：{定义，消除歧义}
- **{术语 B}**：{定义}
- …

## 需求

### 需求 1：{功能名称}（{编号 F-1}）

**用户故事：** 作为 {角色}，我希望 {功能}，以便 {价值}。

#### 验收标准

1. 当 {触发条件} 时，{模块/系统} 应 {预期行为}
2. 当 {触发条件} 时，{模块/系统} 应 {预期行为}
3. 若 {前置条件}，则 {模块/系统} 应 {预期行为}

#### 边界情况

- {边界情况描述} → {预期处理}

### 需求 2：{功能名称}（{编号 F-2}）

…（同上结构）
```

### 格式要点

| 元素 | 规则 |
|------|------|
| 需求编号 | `F-{N}`，从 1 开始递增 |
| AC 编号 | `AC-F{N}-{M}`，内联在验收标准条目前可选标注 |
| EARS 关键词 | 中文：当…时 / 若…则 / 在…期间 / 应 |
| 术语表 | 必须包含。定义项目特有术语和缩写，消除 AI 和新人理解歧义 |
| 边界情况 | 可选。有则写，无则省略整个小节 |
| 用户故事 | 每个需求必须有。格式："作为…我希望…以便…" |

## 实际输出示例：用户认证模块

# 功能需求文档

## 简介

用户认证模块，为系统提供邮箱密码登录、手机验证码登录和密码重置能力。前后端分离架构，需要精确的 API 行为定义作为联调基线。

## 术语表

- **Session**：用户登录后创建的服务端会话，包含 access_token 和 refresh_token
- **access_token**：短期有效的访问令牌（默认 2 小时），用于 API 鉴权
- **refresh_token**：长期有效的刷新令牌（默认 7 天），用于无感续期
- **锁定**：账户因连续密码错误被临时禁止登录的状态

## 需求

### 需求 1：邮箱密码登录（F-1）

**用户故事：** 作为注册用户，我希望用邮箱和密码登录系统，以便访问我的工作空间。

#### 验收标准

1. `AC-F1-1` 当用户提交正确的邮箱和密码时，系统应完成认证、创建 Session、返回 access_token 和 refresh_token
2. `AC-F1-2` 当用户提交的邮箱不存在时，系统应返回统一的"邮箱或密码错误"提示（不泄露邮箱是否注册）
3. `AC-F1-3` 若同一邮箱在 15 分钟内密码错误累计 5 次，则系统应锁定该账户 30 分钟，期间即使密码正确也拒绝登录
4. `AC-F1-4` 当用户勾选"记住我"时，系统应将 refresh_token 有效期从 7 天延长至 30 天

#### 边界情况

- 锁定期间用正确密码尝试 → 仍拒绝，返回剩余锁定时间
- 锁定倒计时结束后，错误计数归零

### 需求 2：手机验证码登录（F-2）

**用户故事：** 作为用户，我希望用手机号和验证码登录，以便在忘记密码时仍能访问账户。

#### 验收标准

1. `AC-F2-1` 当用户请求发送验证码时，系统应生成 6 位数字验证码，通过 SMS 发送，有效期 5 分钟
2. `AC-F2-2` 当用户提交正确的手机号和验证码时，系统应完成认证并创建 Session（同 AC-F1-1 的 token 机制）
3. `AC-F2-3` 若同一手机号在 1 小时内请求验证码超过 5 次，则系统应拒绝发送并提示"请求过于频繁"

#### 边界情况

- 验证码过期后提交 → 返回"验证码已过期，请重新获取"
- 手机号未注册 → 自动创建账户（注册即登录）

### 需求 3：密码重置（F-3）

**用户故事：** 作为忘记密码的用户，我希望通过邮箱验证重置密码，以便重新获得账户访问权限。

#### 验收标准

1. `AC-F3-1` 当用户请求密码重置时，系统应向注册邮箱发送包含重置链接的邮件，链接有效期 30 分钟
2. `AC-F3-2` 当用户通过有效链接提交新密码时，系统应更新密码、使所有现有 Session 失效、要求重新登录
3. `AC-F3-3` 若新密码与最近 3 次使用的密码相同，则系统应拒绝并提示"不能使用最近使用过的密码"

#### 边界情况

- 重置链接被使用一次后再次点击 → 返回"链接已失效"

## 简单任务说明

当任务是简单修复或独立改动时，只需输出**边界收敛摘要**（现有 prd_output_package YAML），不生成结构化需求文档。判断依据：

- 无跨角色协作需求
- 无需行为级验收标准
- in_scope 可以一句话描述完

---

## 模板设计原则

1. **纯 Markdown 输出**：人类可读即机器可读，不使用 YAML 包裹需求内容
2. **术语表必须存在**：即使只有 2-3 个术语，也要显式定义，减少 AI 和人类的理解歧义
3. **AC 整句表达**：每条 AC 是一句完整的话（当…时，系统应…），不拆分为 trigger + expected 两个字段
4. **编号全局唯一**：`AC-F{N}-{M}` 在整个需求文档范围内不重复
5. **边界情况是补充不是必须**：没有边界情况时省略整个小节
6. **向后兼容**：边界收敛摘要（prd_output_package YAML）格式不变，结构化需求文档是额外产物
