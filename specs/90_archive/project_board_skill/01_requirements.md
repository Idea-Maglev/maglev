# 需求定义: project_board_skill

> 文档关系：
> - 上游：`00_intent.md`
> - 下游：`02_design.md`（待创建）

---

## In Scope

1. 新建标准 Skill `project-board`，可部署到所有 Maglev 化项目
2. 动态扫描 `specs/20_evolution/active/` + 相关代码/测试，通过交叉判断确定每个 spec 的真实流程阶段
3. 映射铁三角角色（VO/TP/XG）在每个活跃 spec 上的参与状态
4. 总看板（`specs/20_evolution/board.md`）+ 需求子看板（每个 spec 下的 `status.md`）
5. 生命周期事件驱动更新 + 手动触发
6. 稳定文件结构与更新机制，形成团队使用习惯

## Out of Scope

- 外部系统集成（飞书/Jira/GitLab Board）
- 历史趋势分析或工时统计
- Git commit 级细粒度追踪
- `docs/DASHBOARD.md` 改写（Maglev 自身的静态文档）
- 角色绩效评估
- PM（项目经理）层面的进度管理

## Success Signal

- 在任意 Maglev 化项目中调用此 Skill，能输出一张包含"活跃需求 × 流程阶段 × 角色状态"的看板
- 阶段判断基于 spec 文件 + 代码 + 测试的交叉证据，不依赖手动标注
- 看板持久化到仓库固定路径，每次打开即可看到最新状态

## Key Unknowns

- 人员-角色映射的配置方式（项目级配置文件 vs 从 contribution_log 推断）
- 与现有 Skill 的集成点（reality-sync 是否应在启动时读取看板？crystallization 归档时是否自动更新？）

## 功能需求

### F-1: 活跃需求扫描

> 作为项目成员，我希望看板能自动发现当前所有正在推进的需求，而不需要手动维护列表。

| AC | 描述 |
|----|------|
| AC-F1-1 | 当用户调用项目看板时，系统应扫描 `specs/20_evolution/active/` 目录，识别所有活跃 spec 并列出名称和意图摘要。 |
| AC-F1-2 | 若 `issues/active/` 中存在尚未关联 spec 的独立任务，系统应将其作为"未启动需求"单独展示。 |
| AC-F1-3 | 若 spec 的 `00_intent.md` 中声明了 `## 子需求` 章节，系统应建立父子关联，并在看板中以层级方式展示。 |

### F-2: 流程阶段判断

> 作为项目成员，我希望看到每个需求当前处于主流程的哪个阶段，判断依据是交叉证据而非手动标注。

| AC | 描述 |
|----|------|
| AC-F2-1 | 系统应基于以下证据交叉判断 spec 阶段：spec 文件存在性（00_intent → 01_requirements → 02_design → 03_plan）、代码变更（是否有对应的 skill/code 修改提交）、测试覆盖（是否有对应的测试文件或验证记录）。 |
| AC-F2-2 | 系统应对每个 spec 的 5 个阶段（需求收敛 / 方案设计 / 编码实施 / 综合验证 / 结晶归档）逐阶段评估完成状态，支持叠加态（多阶段同时进行中）。已归档的 spec 不参与看板展示。 |
| AC-F2-3 | 当证据不足以确定阶段时，系统应标记为"阶段待确认"并列出已有/缺失的证据。 |
| AC-F2-4 | 每个 spec 应输出 emoji 进度条（如 ✅→✅→⏳→⬜→⬜），直观展示全流程进展。 |

### F-3: 角色状态映射

> 作为项目成员，我希望看到每个需求中各角色（VO/TP/XG）的当前参与状态。

| AC | 描述 |
|----|------|
| AC-F3-1 | 系统应根据流程阶段推断当前主导角色：需求收敛阶段 → VO 主导；方案设计/编码实施 → TP 主导；综合验证 → XG 主导。 |
| AC-F3-2 | 当项目配置了人员-角色映射时，系统应展示具体人员名称而非仅角色代号。 |
| AC-F3-3 | 每个角色的状态应为：主导中 / 待介入 / 已完成本阶段职责 / 未参与。 |

### F-4: 看板视图输出

> 作为项目成员，我希望看板输出清晰可读的结构化视图，可以快速掌握全局。

| AC | 描述 |
|----|------|
| AC-F4-1 | 系统应输出 Mermaid 流程图，以 subgraph 展示阶段，节点包含需求名称+意图摘要+主导角色。 |
| AC-F4-2 | 系统应输出详情表，包含意图、进度条、主导角色、信心度和导航链接，每列使用人话。 |
| AC-F4-3 | 系统应支持 Markdown 格式输出，兼容终端和 IDE 内阅读。 |
| AC-F4-4 | 看板应包含图例段，解释所有符号和阶段含义。 |

### F-5: 持久化看板文件

> 作为项目成员，我希望看板有一个固定位置的文件，每次打开就能看到最新状态。

| AC | 描述 |
|----|------|
| AC-F5-1 | 系统应将总看板输出写入 `specs/20_evolution/board.md`。 |
| AC-F5-2 | 文件应包含"最后更新时间"标记，让阅读者能判断数据时效性。 |
| AC-F5-3 | 文件格式应保持稳定（表头结构不随内容变化而变），便于团队形成阅读习惯。 |

### F-6: 生命周期事件驱动更新

> 作为项目成员，我希望看板在关键节点自动刷新，而不需要人工记得去更新。

| AC | 描述 |
|----|------|
| AC-F6-1 | 当主流程中的关键生命周期事件发生时（spec 创建、需求通过 Ready Gate、设计完成、实施完成、验证通过、结晶归档），系统应提醒或自动触发看板更新。 |
| AC-F6-2 | 系统应支持手动触发更新（如通过 `/board` 命令或 skill 调用）。 |
| AC-F6-3 | 更新时应对比前后状态，在看板文件中标注"本次变更摘要"（如"spec_X 从方案设计→编码实施"）。 |

### F-7: 需求级子看板

> 作为项目成员，我希望每个活跃需求有独立的细粒度状态视图。

| AC | 描述 |
|----|------|
| AC-F7-1 | 每个活跃 spec 目录下应维护 `status.md`，记录该需求的 AC 完成进度、当前角色参与状态和已知阻塞项。 |
| AC-F7-2 | 子看板应随总看板同步更新。 |
| AC-F7-3 | 当 spec 结晶归档时，子看板随 spec 一起移入 `90_archive/`，不再出现在总看板中。 |

## 术语表

| 术语 | 定义 |
|------|------|
| VO (Value Owner) | 价值定义者，负责需求定义、验收红线和业务决策 |
| TP (Tech Pilot) | 技术领航者，负责架构设计、代码实施和 AI 协作 |
| XG (Experience Guardian) | 体验捍卫者，负责质量验证、交互测试和体验把关 |
| 铁三角 | Maglev 项目的三个核心角色组合（VO + TP + XG） |
| 阶段证据 | 用于交叉判断 spec 所处流程阶段的多源信号（文件存在性 + 代码变更 + 测试覆盖） |
| 总看板 | `specs/20_evolution/board.md`，所有活跃需求的矩阵总览 |
| 子看板 | `specs/20_evolution/active/{spec}/status.md`，单个需求的细粒度状态 |
