# Intent — Spec Provenance Governance

## 背景

在其他项目实践中，spec 生成和修订暴露出两个系统性问题：

1. spec 可能只呈现结论，缺少可回查的来源证据。
2. AI 可能在后续修订中静默改变 spec 语义，读者难以判断为什么变了、依据是什么。

进一步讨论后，问题被收窄为：单纯增加“证据列”仍然不够。AI 可能只抓取局部片段作为证据，而没有阅读完整上下文，导致 AC 片面化、窄化，甚至把孤立语句误判为完整需求。

## 核心意图

为 Maglev 的 spec 生成与修订流程增加“来源依据 + 上下文判定 + 语义变更记录”要求，让执行者和读者都能确认：

- spec 的关键主张来自哪里；
- AC 如何从完整上下文推导出来；
- 来源冲突或后续补充如何被判定；
- AI 修改需求、验收或设计语义时为什么修改。

## 本轮不追求

- 不先实现检查脚本或自动化工具。
- 不批量迁移历史 spec。
- 不改造 `maglev-foresight` 项目本身。
- 不把格式化、错别字、索引维护等非语义编辑纳入变更记录。

## 成功信号

- 新生成的 requirements 明确规定“来源依据”“AC 来源摘要”“AC 上下文判定”“可回查证据”“语义变更记录”的最小要求。
- 需求把复杂度放在执行者应保留的工作痕迹里，只输出读者可消费的信息。
- 需求能直接交给后续方案设计，用于更新模板、流程和审计面。
