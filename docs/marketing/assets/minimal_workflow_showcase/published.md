# 一个最小工作流示例

很多人第一次听到 Maglev，会以为它一定很重。

但如果只看一个最小闭环，它其实可以很简单：

> 先把这次要做的事说清楚，再让实现、验证和后续接手都围绕同一份依据展开。

这篇示例不讲完整体系，只回答一个更实际的问题：

> **如果团队只想先跑通一次最小闭环，Maglev 到底会多做什么，又少掉什么麻烦？**

## 先看一个真实样本

如果借用一个“项目主数据”系统里的真实任务作为样本，
一个很典型、也很适合说明最小闭环的任务会是：

- 签字前新增“当年投资”必填校验
- 发起立项审批时补强风险提示文案和用户手册入口
- 变更流程里支持在线新增临时 T2 子项目

如果你不知道“项目主数据”是什么，可以先把它理解成：

- 一个管理项目基础信息、投资金额、角色分工、审批状态和后续变更的内部系统
- 它不是单纯的表单页，而是很多后续动作的入口，比如签字、立项审批、投资调整、子项目拆分
- 一旦这里的信息不准，后面的审批、财务、协作和执行都可能跟着跑偏

这也不是一个简单的页面改动。
它大致是在处理这样一类问题：

- 什么情况下项目可以进入签字
- 发起立项审批前，系统要提醒用户哪些关键风险
- 投资变更时，哪些改动需要走审批，哪些改动可以在系统内直通处理
- 当一个项目下面要临时新增二级子项目时，这个子项目在“保存、提交、正式生效”之间怎么流转

换句话说，这个任务真正处理的不是一个字段，而是一条业务流程怎么在系统里被稳定承接。

这不是一个超大的需求，但已经足够体现真实研发里的几个难点：

- 同一条业务链路会同时经过签字、审批、变更三个节点
- 变化会同时影响前端交互、后端校验和流程状态
- 这个任务还依赖前置的字段清单和资源池规则改造

如果直接把这类需求丢给 AI，当然也能很快开始写代码。
但问题往往出在后面：

- 前端改了文案，后端校验没补齐
- 后端补了字段，前端草稿和场景分流没跟上
- 新增的 T2 子项目能录入，但生命周期和验收口径没对上
- 最后回头看，大家都做了一部分，但整条流程并没有真正对齐

真正拖慢团队的，通常不是少写几行代码，而是这条链路上的理解不断走散。

如果把这个样本再压缩一下，它其实就是一个很典型的四段式闭环：

1. 先说清楚这次到底改哪条流程
2. 再拆清楚前后端各自要承接什么
3. 再把执行顺序和依赖写明白
4. 最后把验收条件落成可检查的清单

```mermaid
graph LR
    clarify["需求澄清"]
    spec["Spec"]
    implement["实现"]
    validate["验证"]
    retain["沉淀"]

    clarify --> spec
    spec --> implement
    implement --> validate
    validate --> retain
```

这也是 Maglev 在最小闭环里最重要的动作：
不是直接把一句需求扔给实现，而是先把需求澄清收成一份可执行的 Spec，再让实现、验证和后续复用围绕它展开。

## 没有 Maglev 时，这个需求通常怎么推进

很多团队里的真实路径，往往更接近下面这样：

- 先从 PRD 或会议记录里摘几句需求
- AI 先改一个页面或一个方法
- 人再补充“这个规则其实还影响签字和审批”
- 做到后面才发现，还有草稿、场景分流、验收用例没有对齐

短期看好像更快，实际上只是把澄清成本推迟了。

## 用这个真实样本看，最小闭环怎么跑

把这条真实任务线拆开看，流程其实很清楚。

### 第一步：先把“这次到底要改什么”说清楚

在 `01_requirements.md` 里，团队先做的不是开始改代码，
而是把这次变更重新整理成几个明确问题：

- 签字为什么会被拦住
- 发起审批前用户需要被提醒什么
- 变更流的三个场景分别怎么分流
- 临时新增 T2 子项目的生命周期怎么处理

如果把这些问题翻成更业务化的话，其实是在回答：

- 哪些项目信息不完整时，不能继续往下流转
- 哪些投资变动会影响审批和财务口径
- 哪些子项目只是本次变更里的临时内容，哪些会在审批通过后真正落库
- 用户在流程中途退出时，系统要不要保留上下文

这一步的价值很直接：

> **先把“这次到底在改哪条业务链”说清楚。**

### 第二步：把同一个问题拆成前后端都能执行的设计

接下来不是让前后端各自猜，而是进入 `02_design_frontend.md` 和 `02_design_backend.md`。

这里最关键的地方在于：

- 前端设计会明确到哪个弹窗、哪个组件、哪个交互变化要改
- 后端设计会明确到哪个校验链路、哪个字段贯通、哪个策略类要承接
- 同一个任务里的前后端修改，会围绕同一套场景定义展开

这一步的价值是：

> **不是把一个大需求丢给两边分别理解，而是让两边围绕同一份说明展开。**

### 第三步：把执行顺序和依赖摊开

很多团队的问题不是没有设计，而是执行时又散了。

在这个真实样本里，实施计划和日常同步记录正好说明了 Maglev 在这里补了什么：

- 哪些后端字段要先补
- 哪些前端改动可以先做
- 哪些内容依赖前置规则调整
- Spec 里的路径和真实代码路径怎么对应

也就是说，计划不是一句“开始开发”，而是：

> **把谁先做、做到哪里、还缺什么，提前写出来。**

### 第四步：把验收写成可以检查的东西

最后这一步，经常最容易被忽略。

在 `04_test_cases.md` 里，验收不是一句“功能正常”，而是继续拆成：

- 什么情况下签字按钮必须置灰
- 审批弹窗里的链接是否能正确打开
- 草稿是否按场景隔离
- 变更流程里的字段和提示是否一致

这一步的意义在于：

> **团队留下来的不只是实现，还有以后继续回归的依据。**

## Maglev 最小闭环里，具体多了什么

这个最小工作流不要求先写很多文档。
它只要求先补一份足够小、但足够清楚的执行依据。

可以把它理解成下面四步：

| 步骤 | 团队在做什么 | 可见产出 | 它解决什么问题 |
| :--- | :--- | :--- | :--- |
| 固定意图 | 把模糊需求改写成可执行说明 | 一页小型 Spec | 避免每个人理解不同 |
| 执行实现 | 让人和 AI 围绕同一输入推进 | 代码变更、接口变更、页面变更 | 避免只改表面、不改关键约束 |
| 对齐验证 | 按前面的说明检查是否真的完成 | 验收清单、关键测试点 | 避免最后才发现偏差 |
| 沉淀资产 | 把这次共识留下来，供下次继续用 | 可复用规则、变更记录 | 避免下次又从头解释 |

读表结论：
**Maglev 增加的不是“再一层流程”，而是一份让后续动作对齐的依据。**

## 这份最小闭环里，最关键的输入长什么样

它不需要很长，但至少会把下面几件事写清楚：

- 这次改动到底覆盖签字、审批、变更流里的哪些节点
- “当年投资”这个新增字段会影响哪些校验
- 临时 T2 子项目的创建、保存、提交分别发生在哪一段流程里
- 什么情况算完成，什么情况不算

如果压缩成这类场景里的示意，会接近这样：

| 项目 | 示例内容 |
| :--- | :--- |
| 目标 | 让签字、审批、变更流同步适配新增字段和流程规则 |
| 核心规则 | 当年投资必须进入签字校验；场景 1/2/3 的流转边界要明确 |
| 影响范围 | 前端交互、后端校验、草稿与审批流转 |
| 完成标准 | 签字、审批提示、变更场景、测试用例四处口径一致 |
| 不在本次范围 | 不重写整套变更框架，只在现有骨架上做增量改造 |

这里最关键的，不是格式，而是：

> **AI 和人后面的动作，终于不再直接消费一句模糊需求，而是消费一份更清楚的说明。**

## 在真实项目里，这个最小闭环会留下什么痕迹

这不是纸上谈兵。

在一个正在执行 Maglev 的“项目主数据”系统里，这类任务很适合拿来做这个示例。
它讨论的是“签字、立项审批、变更流”这一条真实业务链路，而且不是孤立改动，而是和前后的规则调整相互关联。

围绕这个任务，可以直接看到一条完整的资产链路：

| 阶段 | 仓库里可见的产物 | 它说明了什么 |
| :--- | :--- | :--- |
| 需求收敛 | 一份需求说明 | 模糊目标被整理成具体场景、变化点和验收标准 |
| 设计展开 | 前后端设计说明 | 同一个任务被拆成前后端各自可执行的设计 |
| 实施计划 | 一份实施计划 | 改动被拆成阶段化任务，而不是一句“开始开发” |
| 执行同步 | 日常同步记录 | 每日执行时的目标、依赖和路径映射被显式记录 |
| 验收检查 | 测试用例清单 | 验收不再只靠“感觉差不多” |

读表结论：
**最小闭环真正可见的标志，不是多了一份文档，而是形成了一条从输入到验收都能追溯的链路。**

## 为什么这比“直接让 AI 写”多一步

而 Maglev 的思路是：
先把模糊的部分压缩成团队可共同理解的东西，再交给实现和验证。

这一步看起来多做了一点事，但它换回来的是：

- 更少的“你先写一版我再改口”
- 更少的局部正确、整体跑偏
- 更少的验收争议
- 更容易把这次需求变成下次还能复用的上下文

## 一次最小闭环里，Maglev 具体提供什么

如果只问“它有没有写代码”，会看不见重点。
这类最小闭环里，Maglev 更实际的增量通常是下面这些：

| 环节 | 没有 Maglev 时 | 有 Maglev 时 |
| :--- | :--- | :--- |
| 输入 | 一句模糊需求直接开做 | 先形成小而清楚的 Spec |
| 实现 | 谁先想到什么就先改什么 | 页面、接口、导出围绕同一范围推进 |
| 验收 | 写完再靠印象检查 | 直接对照完成标准检查 |
| 后续修改 | 下次还要重新解释 | 上次的规则和边界可以继续复用 |

读表结论：
**Maglev 真正补上的，是输入质量、执行边界、验收依据和后续复用条件。**

如果继续用这个真实样本来翻译，上面四点可以再说得更具体一点：

| Maglev 提供什么 | 在这个样本里的表现 |
| :--- | :--- |
| 更清楚的输入 | 先明确签字、审批、变更流各自要改什么 |
| 更稳定的边界 | 明确哪些改动在前端，哪些校验在后端，哪些依赖前置规则调整 |
| 更可执行的计划 | 把字段、组件、策略类、阶段顺序写成清单 |
| 更可回归的验收 | 把按钮、弹窗、草稿、场景流转拆成测试项 |

读表结论：
**Maglev 不是让团队“多看几份文档”，而是让同一个任务有了一条能走通的主线。**

## 它和代码生成工具是什么关系

这也是很多人关心的一点。

在这个最小闭环里，代码仍然可以由：

- 工程师自己写
- 现有 AI Coding 工具来写
- 更偏 spec / SDD 驱动的代码生成工具来承接

Maglev 不替代这些工具，但并不是完全中立。

因为它的一个核心价值，就是把模糊需求整理成更清楚的 Spec。
所以越是愿意直接消费 Spec 的代码工具，越能放大 Maglev 的价值。

这点在真实项目里也能看出来。
像“项目主数据”仓库的根目录说明就明确写着：改代码前先看 `specs/`；核心业务逻辑不再只散落在代码里，而是被提取到 `specs/` 目录统一管控。

读表结论：
**Maglev 最适合站在代码生成之前，先把“要做什么”说清楚。**

## 这篇示例想证明什么

它不是要证明 Maglev 已经覆盖所有复杂场景，也不是要把一个真实任务写成大工程。
它想证明的是：

> **即使只跑一个很小的需求闭环，Maglev 也已经在减少“意图、实现、验证”之间的漂移。**
