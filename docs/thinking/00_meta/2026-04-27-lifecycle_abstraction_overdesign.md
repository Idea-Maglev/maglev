# Lifecycle 抽象错位与 3 态修订（2026-04-27）

## 1. 触发

主题 `docs_knowledge_archival_methodology` 收尾后，紧接的姊妹主题 `runtime_distribute_docs_index_protocol` 在 PRD 落盘环节，用户追问"`_drafts/` `_meta/` 这两个目录是做什么的、是谁的产物、闭环如何被驱动"。

我先按"驱动力缺失"角度回答（提议加触发器、加 stale 检查），但用户一句话拆穿：

> "docs 里这么多文档基本是我和 AI 助手单独产出的——总结一次就直接是正式版，根本不会走对外文章撰写那种有草稿有正式发文的正式流程，所以我的疑问是：这个草稿目录如何会被驱动到，因为我根本不会去消费。"

这句话暴露的不是"机制不完善"，是**抽象不匹配**。

## 2. 抽象错位的本质

F1 上一会话照搬 MemoryOS / 长篇撰稿生命周期：

```
draft  →  active  →  crystallized  →  archived
（STM）   （MTM）     （LPM）          （归档）
```

但真实使用模式是：

```
会话发生  →  AI+用户协作  →  会话末/告一段落  →  快照式总结落到正式位段
                                                ↓
                                       已经是"成品"
                                                ↓
                                  后续修正靠"再写一篇 follow-up"
                                  或"红队反思补充"
                                  ——不是修改原稿
```

**`draft` 这个状态在真实使用中从未存在过。**

铁证：撤回前扫描 `docs/thinking/_drafts/` 共 **0 篇**真实草稿（只有上一会话 F1 落地时建的 README）。

## 3. 为什么之前没发现

3 个原因，按"反思的盲区"顺序：

1. **R1 红队反思命中率盲区**：刚做完的东西命中率最低（已被自我修过），存量越久命中率越高。F1 是上一会话刚做的，红队反思自动绕过。
2. **R3 切换审计对象的延迟**：本会话末尾才升到"扫规范"，没人想到反过来扫"刚做完的产物本身是不是过度设计"。
3. **MemoryOS 类比的诱惑力**：4 态映射 STM/MTM/LPM 看着很优雅、很学术、很"有体系感"——这种**抽象的美学价值**会盖过"它是不是真的有用"的检验。

第 3 条最值得记。**优雅的抽象会假装自己是必要的抽象。**

## 5. `_meta/` 不在本次撤回范围

`_drafts/` 和 `_meta/` 性质不同：

- `_drafts/` 是**状态机的一态**——撤回因为状态本身不存在
- `_meta/` 是**机读侧道**（脚本输出）——不属于状态机，独立存在

`_meta/` 仍有一个独立问题：**新鲜度无人维持**——`cognitive_map.py` 不跑就过期，没有定时器/CI hook。这是另一个问题（"工具持续性"），不是本次撤回处理的。

## 6. 这件事教了什么

### 教训 1：骨架先于数据 ≠ 骨架可以脱离数据

上一会话写过 `skeleton_before_data` 这篇 thinking，立场是"先建结构再填内容"。本次发现这话有反面：

> 装好骨架但**抽象错位**，比不装更糟。
> 因为它给人"已经有机制了"的错觉，掩盖了"机制不匹配真实模式"的事实。

### 教训 2：借鉴范式 ≠ 复用范式

MemoryOS 4 态在它原本的语境（多轮长程对话记忆）是合理的——那里有真实的"未确认 vs 已确认"区分。
搬到 docs/ 这种"会话末快照式总结"语境，**那个区分本身就不存在**——所以套用 4 态相当于凭空发明了一个状态。

判别标准（事后总结）：

> 借鉴一个范式时，**先用反证法**：
> - 如果删掉这个范式的某个状态/角色/环节，会发生什么？
> - 如果什么也不会发生，那这个状态/角色/环节就是**装饰**，不是**结构**。

`draft` 状态删掉之后什么也不会发生（因为本来就没人用）→ 它是装饰。

### 教训 3：用户一句话，抵 3 轮红队

```
红队 R1（11 项问题）        — 0 项命中 F1 抽象错位
红队 R2（80/20 重判）       — 0 项命中 F1 抽象错位
红队 R3（规范瘦身）         — 0 项命中 F1 抽象错位
用户："我根本不会去消费"   — 1 句话直接拆穿
```

红队反思的天花板在于**它仍然在"机制能不能更完善"框架内**。
用户视角的天花板在于**"这玩意我会不会用"——这是结构性的、外部的、不可被红队反思自动产生的**。

所以：**红队反思替代不了用户感知验证**。

## 7. 一句话

> 优雅的抽象会假装自己是必要的抽象。判别它是否真的必要，最快的办法是问"我会不会用"。
