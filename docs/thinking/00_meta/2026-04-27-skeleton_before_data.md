# 骨架先于数据：P0 三块的一致策略

> **Date**: 2026-04-27
> **Status**: active
> **Trigger**: docs_knowledge_archival_methodology 主题 P0 主线（F6/F1/F8）三块骨架全部落地后总结

## 1. 现象

P0 主线 commit 序列：

| 块 | Commit | 落地内容 | 真实数据 |
|---|---|---|---|
| F6 归档触发器 | `bd1c653` | `archive_triggers.py`（3 触发器 + apply 路径） | 默认阈值下 0 候选 |
| F1 生命周期 | `2c29549` | `lifecycle.md` + `_drafts/` + L08 校验 + LPM 保护 | 当前所有叶子均 draft |
| F8 认知地图 | `dbe5209` | `cognitive_map.py`（节点+边聚合 + JSON + INDEX 注入） | 9 节点 0 边 |

**共同结构**：每一块都先建**机制**（脚本 + 协议 + 文档），再等**数据**（实例化的状态字段、跨段引用、归档候选）。

## 2. 这不是巧合

三块都不是"先把数据填齐再封装机制"，而是反过来：

1. **机制小、数据大**：脚本一次写完几百行，但要给 107 个叶子补 status 字段是一个长尾任务
2. **机制是单点改动，数据是分布式改动**：作者一个人能写完脚本，但补叶子需要逐文件回看
3. **机制能立即固化决策，数据要慢慢积累**：lifecycle.md 一旦发布就有约束力，但叶子状态一周后才会有 5% 改写

## 3. 为什么这种顺序对

如果**数据先于骨架**：

- 没有 schema，每个人补的字段格式都不一样，等骨架来了还要清洗
- 没有脚本，补完的数据无法被验证，等于无效劳动
- 没有 lifecycle 文档，作者不知道"补 status=active 是什么意思"

如果**骨架先于数据**：

- ✅ 数据格式被 schema 固定，补的时候就是终态
- ✅ 脚本能立即对补完的数据做反馈（图多一条边、stats 加一）
- ✅ 文档让作者理解动作的语义，补完的字段不只是"应付检查"

## 4. 代价：骨架阶段的"假数据风险"

骨架先于数据的代价是：脚本跑出来的结果可能是**空**的、**全 draft 的**、**0 候选的**。

如果这时偷懒造假数据让 demo 好看，会反向腐蚀整个系统（参见 `../10_critique/2026-04-27-f8_empty_graph_as_honest_signal.md`）。

正确做法：**让空结果可见**，把它当成 D2（数据补全）阶段的 KPI baseline。

## 5. 元观察

这条策略不属于本主题独有，是个泛化的**新机制引入秩序**：

| 阶段 | 产出 | 谁能做 |
|---|---|---|
| Phase 1 骨架 | 机制 + 协议 + 文档 | 单点（作者 + AI） |
| Phase 2 数据 | 实例化的字段、关系、候选 | 分布式（所有作者长尾参与） |
| Phase 3 反馈 | 脚本生成的指标 + 图谱 + 触发器输出 | 自动化 |

Phase 1 不能等 Phase 2 完成；Phase 2 必须以 Phase 1 已有为前提；Phase 3 是 Phase 1+Phase 2 的乘积。

## 6. 反例提醒

这条策略的反例：**急着出 demo 时把骨架和假数据一起 ship**。

后果：

- 假数据被当成真相参考，决策被污染
- 真数据进来后要先擦除假数据，多一道工序
- 用户对图谱失去信任（"这个图是真的还是骗我的？"）

宁可 ship 一个空图，不要 ship 一个假图。

## 7. 适用范围

| 适用 | 不适用 |
|---|---|
| 索引、图谱、归档触发器（结构性产物） | 一次性脚本（写完即扔） |
| 长尾数据需要分布式补全 | 数据集中在一两个文件 |
| 数据语义需要规范定义 | 数据本身就是 ad-hoc 的 |
| 早期可以接受空结果 | 有刚性 demo 时间窗口 |
