# 发布说明交付边界：终端提示，不下发到用户项目

> **Trigger**: 更新链路曾把 Maglev 自身的 `CHANGELOG.md` 作为普通分发资产同步到用户项目根目录。
> **Decision**: 发布说明属于发行沟通资产，不属于用户项目运行资产；更新时应在终端展示，而不是落盘到项目工作区。

## 1. 问题是什么

把 `CHANGELOG.md` 同步到用户项目根目录，看起来像是“顺手给用户一份说明”，但它会带来三个副作用：

1. 用户项目根目录多出一个与业务无关的文件
2. AI 助手会把这份上游发布说明误读成当前项目上下文的一部分
3. 更新链路会把“发行沟通资产”和“运行时资产”混成同一类下发物

这不是单纯的目录洁癖问题，而是运行语义污染。

## 2. 为什么它不属于用户项目

用户项目真正需要的是：

- `.agents/`
- `.maglev/`
- workflow / skill / rules / protocols
- 与当前项目协作直接相关的上下文资产

而 `CHANGELOG.md` 解决的是另一件事：

- 告诉用户“这次版本更新了什么”

它的消费时机是：

- `init`
- `update --dry-run`
- `update`

它的消费介质应是：

- 终端输出
- 发布记录归档

而不是项目根目录。

## 3. 正确分层

### A. 运行时资产

应进入 manifest，并同步到用户项目：

- skills
- workflows
- protocols
- rules
- 必要的配置与运行支持文件

### B. 发行沟通资产

不应进入用户项目工作区：

- `CHANGELOG.md`
- 发布说明
- 版本解读文本

它们应存在于：

- release 分支 / 发布镜像
- `docs/releases/`
- update 时的终端输出

## 4. 当前结论

更新链路的正确行为是：

1. manifest 不下发 `CHANGELOG.md`
2. installer 在 `update` / `dry-run` 后读取版本说明
3. 直接在终端打印给用户
4. 用户项目根目录不新增发布说明文件

这样可以同时满足：

- 用户能看到版本变化
- 项目工作区保持纯净
- AI 不会把上游 changelog 当成当前项目事实
