# 一次"先做后反思"的会话复盘：多轮对抗 + 80/20 调校 + 规范瘦身

> **Date**: 2026-04-27
> **Status**: active
> **Trigger**: 单次会话内连续完成 P0 主线、knowledge-check、对抗反思、80/20 重判、规范审计 5 个阶段后

## 1. 会话拓扑

这次会话从 P0 主线推进 → 知识沉淀检查 → **三轮反思** 的非线性结构：

```
推进  →  knowledge-check  →  R1 对抗反思  →  R2 80/20 重判  →  R3 规范审计  →  落地修复
                              (找问题)     (筛真问题)      (换视角)     (收 ROI)
```

每一轮反思都**否决了上一轮的部分结论**：

- R1 对抗反思暴露 6 项残留风险，主张推 D2
- R2 用 80/20 重判 → R1 的 6 项中 5 项是完美主义噪音，D2 是错判
- R3 切换审计对象（从"代码"换到"AI 助手规范"）→ 发现 AGENTS.md 真正可砍的 52 行

## 2. 三轮反思的结构性差异

| 轮次 | 对象 | 立场 | 产出 | 价值 |
|---|---|---|---|---|
| R1 红队对抗 | 本轮自己写的代码 | "找漏洞，不留情面" | 11 项问题清单 | ⭐⭐ |
| R2 80/20 重判 | R1 的 11 项问题 | "哪些值得修，哪些是噪音" | 1 项真问题 + 1 项错判 | ⭐⭐⭐⭐ |
| R3 规范审计 | AGENTS.md + skills 体系 | "堆砌 vs 客观" | 3 项 20% 侧修改 | ⭐⭐⭐⭐⭐ |

**关键观察**：R3 的产出 ROI 远高于 R1 + R2 之和。原因不是 R3 更聪明，是**对象选择更对**：

- R1 对象（自己刚写的代码）→ 产出"我应该做更多"
- R2 对象（R1 的清单）→ 产出"R1 大部分是噪音"
- R3 对象（更早期的存量规范）→ 产出"存量本身有 44% 可压缩"

存量比新产出更适合被审计。新产出的"问题"很多其实是**未来才会出现**的（D2 数据、L09 警告对象、e2e 测试）；存量的问题是**已经在每天发生**（每读 AGENTS.md 重复一遍冗余信息）。

## 3. 自我迭代中的"完美主义引力场"

R1 写的时候，我自己都没意识到偏差 —— 把"自我兑现的诚实信号"挑出来批判，把"L09 警告无对象"列为高优。

**这是工程师本能**：看到代码就找漏洞，看到机制就想加校验。

但工程师本能在**新做完的产出**上失效。理由：

1. 刚做完时，问题密度低（高密度问题已经在做的过程中被修了）
2. 剩下的"问题"多是 long-tail（覆盖率、enforcement、e2e）
3. 这些 long-tail 在数据稀疏时**无对象可作用**（L09 没有 status 字段可警告，因为没人写 status）

**结论**：刚做完一件事时，红队的命中率最低。这时候的红队是仪式性的，不是诊断性的。

应该等待**该事被使用一段时间后**，让真实使用暴露真问题，再做红队。

## 4. R3 的成功路径：换审计对象

当 R1 + R2 没有产出真正高 ROI 的修复时，正确动作不是"再写一轮反思"，是 **切换审计对象的层级**。

层级跳跃：

```
代码（具体）  →  AGENTS.md（规范）  →  catalog 体系（meta-meta）
↑              ↑                   ↑
高细节         中抽象              高抽象
低 ROI         高 ROI              中 ROI
```

R3 选 AGENTS.md 是因为：

- 它是**每次会话都会被注入**的内容（影响面大）
- 它是**很久没改的存量**（堆积概率高）
- 它**不是本轮新做的**（没有自我辩护偏差）

如果 R3 也选了"代码"作为对象，大概率重复 R1 的产出。

## 5. 规范瘦身的 3 个判定标准

R3 删除 AGENTS.md 52 行的判定不是凭感觉，是 3 条客观标准：

| 标准 | 检测方法 | 本案例 |
|---|---|---|
| **重复信息源** | 同一事实写在两处或更多 | 技能列表在 AGENTS.md + private-catalog.yaml + SKILL.md 共三处 |
| **内部矛盾** | 两条规则互相牵制 | "灵感留存"全局原则 vs "knowledge-check"按需触发 |
| **状态过期** | 用了"过渡期"、"暂时"、"待"等词但无终止条件 | "过渡期保留 maglev-librarian"无具体退役日 |

满足任一即可删/改。3 项都满足时**必须**改。

## 6. 这次会话的元产出

按"为什么 / 是什么 / 我现在该怎么做"拆：

| 维度 | 内容 |
|---|---|
| 为什么这么做 | 多轮反思发现"刚做完的产出不适合做红队"、"存量更适合审计" |
| 是什么 | 一种"3 轮反思 + 切换对象层级"的自我调校模式 |
| 该怎么做 | 下次推完一个主线后，先 knowledge-check（沉淀），再 80/20 验收（不要挑边际），最后审计存量规范文件（最高 ROI） |

## 7. 反例：如果跳过 R2 + R3

| 没跳过 | 跳过 |
|---|---|
| 落地 8 commits | 落地 7 commits + 1 篇过度反思 |
| 真实修复 AGENTS.md -52 行 | 写一份"还需做的 6 件事"待办 |
| 后续会话进入时减少 52 行噪音 | 后续会话还是读到冗余指令 |
| 完美主义被识别后剥离 | 完美主义伪装成"自我严格" |

## 8. 适用范围

| 适用 | 不适用 |
|---|---|
| 多轮自我对抗的会话 | 一次性输出（写完就结束） |
| 有存量规范可审计 | 全新项目，无存量 |
| 用户允许"切换对象层级" | 用户严格按既定路径推进 |
| 工作模式有完美主义倾向 | 工作模式已经倾向于过度简化 |

## 9. 一句话总结

> 红队找问题的命中率，跟审计对象的"年龄"成正比；跟"刚做完"的距离成反比。

刚做完的东西问题最少（已被自我修过一遍），最不值得红队；
存量越久的东西问题最多（堆积 + 状态过期），最值得红队。

这条原则在工程实践中被反复验证（legacy code review 比新代码 review 找出更多问题）；本次会话只是把它在**会话内**复现了一次。
