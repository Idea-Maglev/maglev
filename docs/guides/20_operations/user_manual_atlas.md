# Maglev Atlas：用户导航手册

> 目标：帮助已经接入 Maglev 的用户理解“安装后通常怎么找方向、怎么开始一轮会话、怎么判断下一步”。

## 1. 先说适用范围

这篇文档更适合下面这类场景：

- 你已经把 Maglev 接入了项目
- 你想知道安装后通常先做什么
- 你不想记太多技能名，只想知道常见入口和使用节奏

## 2. 安装后最常见的几个入口

### `standup`

适合：

- 每轮会话刚开始时
- 你需要快速同步当前主线、风险和下一步动作时

它通常会帮助你：

- 识别当前主要任务
- 同步最近的上下文
- 输出 `Space / Mind / Risk / Action / Mode`

### `map`

适合：

- 你刚进入一个项目，想先看结构
- 你需要快速理解当前仓库有什么内容

它通常会帮助你：

- 扫描项目结构
- 更新或生成项目地图
- 帮你更快找到接下来该看的文件

### 面向具体任务的能力

当你已经知道自己要做什么时，再进入更具体的能力，例如：

- `create-spec`
- `reverse-spec`
- `quick-dev`
- `validate-all`

你不需要一次记住所有能力，通常只需要先知道：

- `standup` 用来开场同步
- `map` 用来看结构
- 具体任务再选对应能力

## 3. 一个更自然的使用顺序

在已经接入 Maglev 的项目里，更常见的顺序通常是：

1. 先用 `standup` 同步当前上下文
2. 如果还不熟结构，再用 `map`
3. 根据任务选择 `create-spec`、`reverse-spec`、`quick-dev` 等能力
4. 交付前再用验证或审计能力收口

如果你更关心这些入口彼此是什么关系，可以继续看：

- [多入口使用说明](./maglev_entrypoints.md)
- [角色与流程翻译](./maglev_role_flow_translation.md)

## 4. 使用时的几个判断原则

### 先安装，再期待本地工作流

对一个还没接入 Maglev 的全新项目，不要默认 `/standup`、`/init` 这些本地工作流已经存在。

### 不熟项目时，先同步上下文

如果你一上来就直接做实现，后面更容易跑偏。先做一次 `standup`，通常更稳。

### 不确定下一步时，优先看地图和当前任务

Maglev 更适合“先看当前状态，再决定动作”，而不是死记很多技能名。

如果你更想按“产品 / 开发 / 测试 / 技术负责人”这些传统角色来理解主流程，可以继续看：
